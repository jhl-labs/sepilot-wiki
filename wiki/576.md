## 개요
- **문서 목적**: 프로덕션 환경에서 대형 언어 모델(LLM)의 출력 변화를 지속적으로 감시하고, 비즈니스에 미치는 영향을 최소화하기 위한 관측 전략을 제공한다.  
- **대상 독자**: ML 엔지니어, SRE·DevOps 팀, 제품 매니저, AI 서비스 운영 담당자  
- **필요성**: 모델·프롬프트는 동일하지만 서비스 제공자가 백그라운드에서 가중치를 업데이트하거나 데이터 분포가 변하면서 출력이 달라지는 **LLM 드리프트**가 빈번히 발생한다. 드리프트가 감지되지 않으면 스팸 필터 오작동, 데이터 추출 오류, 코드 품질 저하 등 비즈니스 위험이 커진다.  

## LLM Drift 정의와 원인
- **드리프트 개념**: 모델이나 프롬프트를 변경하지 않았음에도 불구하고, 시간 경과에 따라 동일 입력에 대한 출력이 달라지는 현상이다. 모델 자체는 동일하지만 출력이 변한다.  
- **주요 원인**  
  1. **서비스 제공자의 백그라운드 가중치 업데이트** – 모델 제공업체가 주기적으로 파라미터를 조정한다.  
  2. **입력 컨텍스트·데이터 분포 변화** – 프로덕션에서 사용되는 질문·요청 패턴이 변하면서 모델 응답이 달라진다.  
  3. **파인‑튜닝·버전 업그레이드에 따른 품질 저하** – 내부 파인‑튜닝이 의도치 않은 성능 저하를 초래할 수 있다.  
- **드리프트가 발생하기 쉬운 시나리오**  
  - 실시간 챗봇에서 최신 뉴스 질문이 급증할 때  
  - 정기적인 데이터 파이프라인 업데이트 후 동일 프롬프트 실행  
  - 모델 제공자가 “안정화” 업데이트를 배포했지만 기존 엔드포인트를 재배포하지 않은 경우  

## 모니터링 지표·메트릭
| 카테고리 | 지표 | 설명 | 참고 임계값 |
|----------|------|------|-------------|
| **출력 유사도** | 코사인 유사도 | 현재 출력 임베딩과 기준(baseline) 임베딩 간 코사인 거리 | 0.8 이하 → 경고 |
| | 토큰 레벨 차이 | 출력 토큰 수 차이 비율 | 20 % 초과 시 주의 |
| | BLEU / ROUGE | 생성 텍스트 품질 비교 (특히 요약·번역) | BLEU < 0.6 시 알림 |
| **드리프트 빈도·심각도** | 드리프트 비율 | 전체 엔드포인트 중 드리프트 감지된 비율 | 10 % 초과 시 SLO 위반 검토 |
| | 평균 감지 시간 | 첫 드리프트까지 걸린 평균 일수 | 30 일 이내 감지 시 빠른 대응 필요 |
| | 심각도 레벨 | 낮음·중간·높음 (코사인 거리·작업 유형에 따라) | 고심각도(> 0.3 코사인 거리) → 즉시 조치 |

> **출처**: 300개 체크 분석 결과에 따르면 전체 엔드포인트의 **23 %**가 30 일 이내에 측정 가능한 드리프트를 보였으며, **8 %**가 유의미한 드리프트(코사인 거리 > 0.3)를 나타냈다[[출처](https://euno.news/posts/ko/i-analyzed-300-llm-drift-checks-heres-what-i-found-d3a2c1)].

### 서비스 수준 목표(SLO)와 알림 기준
- **SLO 예시**: “드리프트 비율이 5 %를 초과하지 않는다.”  
- **알림 트리거**:  
  1. 코사인 유사도 ≤ 0.8 → `warning` 레벨 알림  
  2. 코사인 유사도 ≤ 0.6 → `critical` 레벨 알림 및 Incident 생성  

## OpenTelemetry·Prometheus 기반 구현 예시
### 전체 아키텍처 흐름
1. **데이터 수집** – 프로덕션 API 호출 결과를 로그/스트림으로 수집.  
2. **임베딩 생성** – 동일 입력에 대해 현재 출력과 기준 출력을 임베딩(예: OpenAI Embeddings)으로 변환.  
3. **메트릭 계산** – 코사인 유사도 등 지표를 실시간 계산.  
4. **Exporter** – 계산된 메트릭을 OpenTelemetry SDK를 통해 Prometheus Exporter로 전송.  

### OpenTelemetry Instrumentation (Python 예시)
```python
from opentelemetry import metrics
from opentelemetry.sdk.metrics import MeterProvider
from opentelemetry.exporter.prometheus import PrometheusMetricsExporter

meter = metrics.get_meter(__name__, provider=MeterProvider())
drift_gauge = meter.create_observable_gauge(
    name="llm_drift_cosine_similarity",
    description="Cosine similarity between current and baseline outputs",
    unit="1",
    callbacks=[lambda options: compute_cosine_similarity()],
)

exporter = PrometheusMetricsExporter(port=9464)
exporter.start_http_server()
```

### Prometheus Exporter 설계 포인트
- **메트릭 이름**: `llm_drift_cosine_similarity`  
- **라벨링 전략**: `model`, `endpoint`, `task_type` (e.g., `classification`, `extraction`)  
- **Alertmanager 연동**:  
  ```yaml
  groups:
    - name: llm-drift-alerts
      rules:
        - alert: LLMDriftWarning
          expr: llm_drift_cosine_similarity < 0.8
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "LLM drift detected for {{ $labels.endpoint }}"
            description: "Cosine similarity dropped to {{ $value }}."
  ```

### 배포 팁
- **Sidecar 방식**: 모델 서비스와 동일 네임스페이스에 OpenTelemetry Collector sidecar를 배치해 최소 침투 방식 구현.  
- **샘플링**: 고빈도 엔드포인트는 1분 간격, 저빈도는 5분 간격으로 샘플링하여 비용 절감.  

## 300개 체크 사례 분석 결과
### 데이터셋 개요
- **기간**: 6개월  
- **모델**: GPT‑4, GPT‑3.5, Claude 2, Claude 3  
- **사용 사례**: 분류, 구조화된 추출, 텍스트 생성, 코드 생성, 다단계 추론  
- **샘플 수**: 300개 엔드포인트  

### 드리프트 발생 빈도 요약
- **전체 엔드포인트 중 23 %**가 30일 이내에 측정 가능한 드리프트 감지  
- **8 %**가 유의미한 드리프트(코사인 거리 > 0.3) 발생  

### 작업 유형별 드리프트 차이
| 작업 유형 | 드리프트 비율 | 평균 심각도 |
|----------|--------------|-------------|
| 분류 | 31 % | 낮음‑중간 |
| 추출 | 24 % | 중간 |
| 생성 | 18 % | 낮음 |
| 코드 생성 | 12 % | 낮음 |
| 추론 | 28 % | 중간‑높음 |

> **인사이트**: 분류 작업이 가장 높은 드리프트 비율을 보이며, 이는 미묘한 패턴 인식에 민감하기 때문이다[[출처](https://euno.news/posts/ko/i-analyzed-300-llm-drift-checks-heres-what-i-found-d3a2c1)].

### 모델별 안정성 비교
| 모델 | 드리프트 비율 | 첫 드리프트까지 평균 시간 |
|------|--------------|--------------------------|
| GPT‑4 | 48 % | 45 일 |
| GPT‑3.5 | 22 % | 12 일 |
| Claude 2 | 18 % | 28 일 |
| Claude 3 | 36 % | 60 일 |

> **인사이트**: 최신 모델(GPT‑4, Claude 3)일수록 드리프트 발생이 늦고 비율이 낮다. 오래된 모델은 빠르게 드리프트가 나타난다[[출처](https://euno.news/posts/ko/i-analyzed-300-llm-drift-checks-heres-what-i-found-d3a2c1)].

### 비즈니스 영향이 큰 경우
- **분류 결정**: 스팸 필터가 정상 메일을 차단 → 고객 불만 증가  
- **데이터 추출**: 인보이스 필드 누락 → downstream 시스템 오류  
- **품질 게이트**: 코드 리뷰 AI가 취약 코드를 승인 → 보안 리스크  

## 알림·대응 전략
### 실시간 알림 파이프라인 설계
1. **Prometheus Alertmanager** → **Slack/Email/Webhook**  
2. **Incident Management Tool** (예: PagerDuty, Jira Service Management) 에 티켓 자동 생성  
3. **대시보드**: Grafana에 `llm_drift_cosine_similarity` 그래프와 라벨별 히트맵 제공  

### 드리프트 대응 워크플로우
1. **Baseline 업데이트** – 새로운 정상 출력을 기준으로 재설정 (가장 흔한 방법)  
2. **프롬프트 조정** – 명시적 제약조건·예시 추가  
3. **모델 교체·버전 롤백** – 안정적인 모델(GPT‑4 등)으로 전환 (비용이 가장 많이 듦)  

### 자동화 옵션 및 비용‑효과 분석
- **자동 Baseline 재학습**: 매주 기준 출력 재생성 → 인프라 비용은 최소 (CPU 1 vCPU, 메모리 2 GB)  
- **알림 자동 티켓 생성**: 인시던트 비용 절감 (평균 30 분 빠른 대응)  
- **모델 교체 자동화**: CI/CD 파이프라인에 `model-switch` 스크립트 포함 → 롤백 시간 5 분 이하

| **CI/CD 통합** | 파이프라인 | 테스트 단계에서 `drift-test` 실행, PR 실패 조건에 포함 |

- **CI/CD에 드리프트 테스트 통합**: PR 빌드 시 기준 입력에 대한 현재 모델 출력과 코사인 유사도 검증, 임계값 이하이면 빌드 실패.  
- **Three‑Way Observability**: 로그·메트릭·트레이스를 모두 수집해 원인 분석 시간을 40 % 단축.  

## 결론 및 향후 과제
- **현황**: 300개 엔드포인트 분석 결과, 드리프트는 흔히 발생하며 특히 분류·추출 작업에서 비즈니스 위험이 크다.  
- **한계**: 현재는 코사인 유사도 기반 단일 지표에 의존하고 있어, 복합적인 의미 변화(예: 답변 스타일 변동)를 포착하지 못한다.  
- **향후 연구**:  
  1. **다중 지표 결합** – BLEU·ROUGE·Semantic Similarity를 종합하는 스코어링 모델 개발  
  2. **베이스라인 자동 진화** – 시계열 베이즈 모델을 활용해 정상 범위 동적 업데이트  
  3. **표준화 활동** – LLM Drift 관측을 위한 OpenTelemetry 규격 제안 및 커뮤니티 협업
