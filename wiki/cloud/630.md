## 문서 개요
- **목적**: Amazon Route 53에서 퍼블릭 Hosted Zone을 생성하고 도메인 DNS 레코드를 관리하는 전체 흐름을 단계별로 제공한다.  
- **대상 독자**: AWS를 처음 사용하는 개발자·운영자, 기존 DNS 서비스를 Route 53으로 이전하려는 팀, 클라우드 인프라를 자동화하고자 하는 엔지니어.  
- **범위**: 퍼블릭 Hosted Zone 생성, 기본 레코드 검토, 레코드 추가, 도메인 등록기관에서 네임서버 교체, 구성 테스트까지. 프라이빗 Hosted Zone 및 VPC 연동은 다루지 않는다.  
- **사전 요구 사항**  
  - AWS 계정 보유 및 콘솔 접근 권한  
  - 관리하려는 도메인에 대한 등록기관 계정 접근 권한  
  - 최소 권한 원칙에 따라 `route53:CreateHostedZone`, `route53:ChangeResourceRecordSets` 등 IAM 정책이 부여된 사용자 또는 역할  

## Route 53 및 Hosted Zone 기본 개념
- **DNS와 Route 53의 역할**: DNS는 도메인 이름 ↔ IP 주소 매핑을 제공한다. Amazon Route 53은 AWS가 제공하는 고가용성 DNS 서비스로, 전 세계에 분산된 네임서버를 통해 쿼리를 처리한다.  
- **Hosted Zone이란**: DNS 레코드의 컨테이너이며, 특정 도메인(예: `example.com`)에 대한 모든 레코드(NS, SOA, A, CNAME 등)를 저장한다.  
- **Public vs Private Hosted Zone**  
  - *Public* – 인터넷 사용자에게 DNS 응답을 제공한다.  
  - *Private* – 지정된 Amazon VPC 내부에서만 DNS 응답을 제공한다(본 가이드에서는 다루지 않음).  

## 사전 준비
1. **AWS 계정 및 IAM 권한 확인**  
   - 콘솔 로그인에 사용할 IAM 사용자/역할에 `AmazonRoute53FullAccess` 혹은 최소 권한 정책(`route53:CreateHostedZone`, `route53:ListHostedZones`, `route53:ChangeResourceRecordSets`)이 포함되어 있어야 함.  
2. **도메인 등록 현황 파악**  
   - 현재 도메인이 어느 등록기관에 등록돼 있는지 확인하고, 해당 콘솔에 로그인할 수 있는 계정을 준비한다.  
3. **VPC 확인(선택 사항)**  
   - 프라이빗 Hosted Zone을 사용할 경우 VPC ID가 필요하지만, 퍼블릭 호스티드 존에서는 필요하지 않다.  

## 퍼블릭 Hosted Zone 생성 절차
### 1. AWS Management Console 로그인
- **접속 경로**: https://aws.amazon.com/console/ → `Sign In` → IAM 사용자 또는 루트 계정으로 로그인.  
- **필요 IAM 정책 요약**: `route53:CreateHostedZone`, `route53:ListHostedZones`, `route53:ChangeResourceRecordSets` 등.  

### 2. Route 53 콘솔에서 Hosted Zones 선택
- 콘솔 상단 검색창에 **“Route 53”** 입력 → 서비스 선택.  
- 좌측 메뉴에서 **“Hosted zones”** 를 클릭한다.  

### 3. 도메인 상세 입력
- **Create hosted zone** 버튼 클릭.  
- **Domain name** 필드에 관리하려는 도메인 입력(예: `example.com`).  
- **Type** 옵션에서 **Public hosted zone** 선택(프라이빗은 여기서 제외).  
- **Create hosted zone** 를 클릭하면 AWS가 자동으로 NS와 SOA 레코드를 생성한다.  

### 4. 기본 레코드 검토
- 생성된 호스티드 존 상세 화면에서 **NS (Name Server)** 레코드와 **SOA (Start of Authority)** 레코드가 자동으로 표시된다.  
- NS 레코드에는 AWS가 할당한 네임서버 네 개가 포함되며, 이 값이 도메인 등록기관에 설정되어야 도메인 위임이 완료된다.  

### 5. DNS 레코드 추가
- **Create record** 버튼 클릭 → 레코드 생성 화면 진입.  
- **Record name**: 서브도메인 입력(예: `www`). 비워두면 루트 도메인(`example.com`)이 대상이 된다.  
- **Record type**: A, CNAME, MX 등 필요에 따라 선택.  
- **Value**: 대상 IP 주소 또는 CNAME 값 입력.  
- **TTL (seconds)**: 기본값(300) 사용 가능하지만, 필요에 따라 조정한다.  
- **Create records** 로 저장한다.  

### 6. 등록기관에 네임서버 업데이트
1. 호스티드 존 상세 화면에서 **NS 레코드**에 표시된 네임서버 네 개를 복사한다.  
2. 도메인 등록기관 콘솔에 로그인한다(예: GoDaddy, Namecheap, Naver Cloud 등).  
3. 현재 설정된 네임서버를 **AWS 제공 네임서버**(복사한 값)로 교체한다.  
4. 변경 사항 저장 후, DNS 위임이 완료되면 도메인 쿼리가 Route 53으로 라우팅된다.  

### 7. 구성 테스트
- **전파 확인**: `dig @8.8.8.8 example.com` 혹은 `nslookup example.com 8.8.8.8` 명령으로 NS 레코드가 AWS 네임서버를 가리키는지 확인한다.  
- **웹 브라우저 검증**: `www.example.com`에 접속해 기대한 IP 주소로 라우팅되는지 확인한다. 전파 시간은 몇 분에서 몇 시간까지 소요될 수 있다.

## 추가 설정 및 고급 기능
- **레코드 TTL 최적화**: 자주 변경되는 레코드는 짧은 TTL(60~300초)로, 정적 레코드는 긴 TTL(86400초)로 설정해 쿼리 비용을 절감한다.  
- **Alias 레코드**: A 레코드 대신 Alias 레코드를 사용하면 ELB, CloudFront, S3 정적 웹사이트 등 AWS 리소스를 직접 가리킬 수 있어 별도 IP 관리가 필요 없다.  
- **트래픽 라우팅 정책**  
  - *Weighted*: 여러 엔드포인트에 가중치를 부여해 트래픽을 분산.  
  - *Latency*: 사용자와 가장 가까운 리전으로 라우팅.  
  - *Geolocation*: 국가·대륙 별로 라우팅 규칙 정의.  

## 모니터링 및 관리
- **Route 53 대시보드**: 호스티드 존별 레코드 현황, 쿼리 통계 확인 가능.  
- **CloudWatch 연계**: `Route53ResolverQueryLog` 로그 그룹을 통해 DNS 쿼리 로그를 수집하고, `AWS/Route53` 메트릭(쿼리 수, 오류율 등)을 모니터링한다.  
- **레코드 변경 이력 관리**: 콘솔에서는 변경 기록을 직접 제공하지 않으므로, IaC(Terraform, CloudFormation)와 GitOps를 활용해 버전 관리하는 것이 권장된다.  

## 일반적인 문제 해결 가이드
| 문제 | 원인 | 해결 방법 |
|------|------|----------|
| 네임서버 전파 지연 | DNS 캐시, TTL 설정 | 전파가 완료될 때까지 기다리며 `dig` 로 확인. 필요 시 TTL을 낮게 설정 후 재시도 |
| 레코드 오타·포맷 오류 | 잘못된 레코드 이름·값 입력 | 콘솔에서 레코드 상세 화면 확인, `dig` 로 레코드 조회 후 수정 |
| Private Hosted Zone과 충돌 | 동일 도메인에 퍼블릭·프라이빗 존 중복 생성 | 퍼블릭 호스티드 존만 사용하거나, 프라이빗은 별도 VPC에만 연결 |

## 베스트 프랙티스 요약
- **보안**: 최소 권한 원칙에 따라 IAM 정책을 제한하고, Route 53 API 호출에 MFA를 적용한다.  
- **가용성**: 다중 레코드(예: `www`에 여러 A 레코드)와 라우팅 정책을 활용해 장애 시 자동 페일오버 구현.  
- **비용 관리**: TTL을 적절히 조정하고, 불필요한 레코드(특히 테스트용)를 삭제해 쿼리 비용을 최소화한다.
