## 요약
이 가이드는 **문서 업데이트 감지 잡**(`detect-doc-updates`)이 어떻게 구현되고, CI/CD 파이프라인에 연동되는지를 한눈에 보여줍니다.  
핵심 포인트는 다음과 같습니다.

| # | 핵심 내용 |
|---|-----------|
| 1 | GitHub 레포지토리·S3 스토리지에서 파일 해시를 계산하고 Redis에 저장 |
| 2 | 변경이 감지되면 정의된 **Kafka 토픽 `doc-updates`**에 JSON 스키마 메시지를 전송 |
| 3 | Kafka 토픽을 구독하는 downstream 파이프라인(`build-docs.yml`)이 자동으로 빌드·배포를 트리거 |
| 4 | Helm‑Chart와 GitHub Actions를 이용한 CI/CD 배포 파이프라인 제공 |
| 5 | Prometheus 메트릭, Grafana 대시보드, Slack 알림을 통한 운영 가시성 확보 |

---

## 개요
- **문서 업데이트 감지 작업**은 레포지토리·스토리지에 저장된 문서 파일의 변동을 자동으로 감지하고, 이를 기반으로 CI/CD 파이프라인의 트리거를 발생시키는 핵심 백그라운드 잡입니다.  
- 구현 파일: `lib/scheduler/jobs/detect-doc-updates.ts`  
- 흐름: **Scheduler → Worker → Event Broker(Kafka)**  

### 대상 독자
플랫폼 운영자, DevOps 엔지니어, 백엔드 개발자  

### 기대 효과
수동 검증 비용 절감, 변경 누락 방지, 운영 가용성 및 가시성 향상  

---

## 배경 및 필요성
- 문서 변동을 자동으로 감지하면 **운영 안정성**을 크게 높일 수 있습니다. 이는 AWS Well‑Architected 프레임워크 *Operational Excellence* pillar에서 강조하는 “버전 관리 사용·피드백 루프 구현” 원칙과 일치합니다[[AWS Well‑Architected Operational Excellence Pillar](https://docs.aws.amazon.com/ko_kr/wellarchitected/latest/operational-excellence-pillar/wellarchitected-operational-excellence-pillar.pdf)].  
- 기존 **수동 검증** 방식은 변경 누락·인적 오류 위험이 크고, 배포 주기가 길어지는 단점이 있습니다.  
- 자동 감지는 **CI/CD 자동화**와 결합돼 빠른 피드백 루프를 제공하며, 문서‑as‑code 흐름을 지원합니다.

## 아키텍처 개요
![Architecture Diagram](assets/architecture-detect-doc-updates.png)

- **파일 위치**: `lib/scheduler/jobs/detect-doc-updates.ts`  
- **주요 모듈**  
  - **Scheduler**: cron 표현식 기반 트리거 (예: `0 */5 * * *`)  
  - **Worker**: 파일 스캔·해시 계산·Redis 저장 로직  
  - **Event Broker**: Kafka 토픽 `doc-updates`에 변경 이벤트 전송  

- **외부 의존성**  
  - GitHub API (레포지토리 메타데이터)  
  - S3 (문서 스토리지)  
  - Redis (해시 영구 저장)  
  - Kafka (이벤트 전송)  

### Kafka 토픽 & 스키마
- **토픽명**: `doc-updates` (replication factor 3, partition 6)  
- **Message Key**: `filePath` (예: `docs/guide.md`)  
- **Message Value (JSON)**  

    {
        "filePath": "docs/guide.md",
        "eventType": "modified",   // added | modified | deleted
        "hash": "a3f5c9…",
        "timestamp": "2026-03-15T12:00:00Z",
        "author": "github_user"
    }

### Redis 키 구조
- **키**: `doc:update:{filePath}` → SHA256 해시 문자열  
- **TTL**: 30일 (변경 이력 보관)

## 설정 방법
- **기본 설정 파일**: `detect-doc-updates.config.ts`  

       export default {
         watchPaths: ['docs/**/*.md', 'README.md'],
         cronExpression: '0 */5 * * *', // 5분마다 실행
         kafkaTopic: 'doc-updates',
         redisKeyPrefix: 'doc:update:',
         notification: {
           slackWebhook: process.env.SLACK_WEBHOOK,
           email: process.env.NOTIFY_EMAIL,
         },
       };

- **주요 파라미터**

|----------|------|
| `watchPaths` | 감시 대상 파일·디렉터리 glob 패턴 |

| `kafkaTopic` | 변경 이벤트 전송 토픽 (`doc-updates`) |
| `redisKeyPrefix` | Redis 키 접두사 (`doc:update:`) |
| `notification` | 알림 채널 (Slack, 이메일 등) |

## 동작 흐름
1. **Scheduler 트리거** → Cron에 따라 `detect-doc-updates` 잡 실행  
2. **파일 스캔** → `watchPaths`를 순회, 각 파일 SHA256 해시 계산  
3. **해시 저장/비교** → 현재 해시와 Redis에 저장된 이전 해시(`doc:update:{filePath}`) 비교  
4. **변경점 검출** → `added`, `modified`, `deleted` 이벤트 유형 결정  
5. **Kafka 전송** → 위에서 정의한 JSON 스키마를 `doc-updates` 토픽에 전송  
6. **후속 파이프라인 연계** → `build-docs.yml`이 토픽을 구독해 자동 빌드·배포 트리거

## 주요 로직 상세
- **파일 해시/버전 비교**  

       const crypto = require('crypto');
       const hash = crypto.createHash('sha256').update(fileContent).digest('hex');
       const redisKey = `${config.redisKeyPrefix}${filePath}`;
       const prevHash = await redis.get(redisKey);
       if (hash !== prevHash) {
         // 변경 감지 로직
       }

- **변경 감지 기준**  

| 유형 | 조건 |
|------|------|
| 추가 | `prevHash`가 없고 현재 `hash` 존재 |
| 수정 | `prevHash`와 `hash`가 다름 |
| 삭제 | 현재 파일이 없고 `prevHash`가 존재 |

- **중복 감지 방지**  
  - 동일 파일에 대해 연속 실행 시 동일 해시가 나오면 이벤트 억제  
  - Kafka `messageId`에 `filePath:hash` 조합 사용  

- **오류 복구·재시도 정책**  
  - 파일 I/O 오류 → `retry` 라이브러리(`retry@0.13.0`)로 3회 재시도, 지수 백오프 적용  
  - Kafka 전송 실패 → `kafka-node` 내장 `retry` 옵션 사용

## 에러 처리 및 트러블슈팅
| 오류 시나리오 | 원인 | 대응 방안 |
|---------------|------|----------|
| GitHub API 403 | 토큰 권한 부족 | `repo:read` 권한 확인, 토큰 재발급 |
| 파일 I/O Timeout | 스토리지 접근 지연 | 재시도 횟수 증가, 타임아웃 설정(`fs.promises.readFile` 옵션) |
| Kafka 전송 실패 | 브로커 연결 불가 | `KAFKA_BROKERS`·`KAFKA_SASL_*` 환경 변수 확인, 네트워크 정책 검토 |
| Redis 연결 오류 | 인증/네트워크 문제 | `REDIS_URL` 및 비밀번호 확인, 서비스 가용성 점검 |

- **재시도·백오프 전략**  
  - `retry-attempts=3`, `initial-delay=2000ms`, `factor=2`  

- **디버깅 팁**  

  - 로컬 dry‑run 실행  

        npm run detect-doc-updates -- --dry-run --config ./detect-doc-updates.config.ts  

  - 로그 레벨을 `debug` 로 설정하면 파일 스캔·해시·Kafka 전송 상세 흐름 확인 가능

## 테스트 및 검증
- **유닛 테스트 (Jest)**  

| 대상 함수 | 테스트 내용 |
|-----------|-------------|
| `computeFileHash()` | 다양한 파일 크기에 대한 SHA256 결과 검증 |
| `compareHashes()` | `added`, `modified`, `deleted` 시나리오 검증 |
| `publishChangeEvent()` | Kafka 프로듀서 모킹을 통한 메시지 포맷 검증 |

- **통합 테스트 시나리오**  

  1. 테스트 레포지토리에서 파일 수정 → 스케줄러 실행 → `doc-updates` 토픽에 이벤트 존재 확인  
  2. 파일 삭제 → 동일 흐름 검증  

- **CI 파이프라인 자동화**  

       - name: Test detect-doc-updates
         run: npm run test:detect-doc-updates

- **테스트 데이터**: `test/fixtures/docs/` 디렉터리에 샘플 마크다운 파일 포함

## 보안 및 권한 관리
- **시크릿 저장·전달**  
  - GitHub 토큰, Kafka SASL 인증, Redis 비밀번호는 Kubernetes `Secret` 또는 HashiCorp Vault에 저장  
  - 컨테이너 시작 시 `envFrom`으로 주입  

- **최소 권한 원칙**  
  - GitHub 토큰: `repo:read`  
  - S3: `s3:GetObject`, `s3:PutObject` (필요한 버킷만)  
  - Redis: `ACL`을 이용해 읽기·쓰기 전용 사용자 생성  

- **감사 로그**  
  - 모든 API 호출·Kafka 전송에 `requestId` 포함  
  - Loki 혹은 Elastic Stack으로 중앙집중식 로그 전송
