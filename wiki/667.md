## 1. 개요
자동 일기 에이전트는 **cron job**을 트리거로 하여 매일(또는 매주 평일) 최신 메모리 파일을 읽고, 빌드‑로그 항목을 작성한 뒤 GitHub에 커밋하고, dev.to에 초안을 생성·게시하며, 최종 결과를 Telegram으로 알리는 완전 자동화 파이프라인입니다. 인간의 개입 없이 전 과정을 수행함으로써 **일관성**, **효율성**, **책임 분리**를 달성합니다.

주요 구성 요소  
- **cron job** – OpenClaw이 제공하는 스케줄러  
- **OpenClaw** – 작업 정의와 API 호출을 담당하는 에이전트 프레임워크  
- **GitHub** – 일기·빌드 로그 저장소  
- **dev.to** – 자동 포스팅 대상 블로그 플랫폼  
- **Telegram Bot** – 알림 전송 채널  

> 내용은 EUNO.NEWS 기사 ["자신의 일기를 쓰는 에이전트 — 자동으로"](https://euno.news/posts/ko/the-agent-that-writes-its-own-diary-automatically-910f8a) 를 기반으로 작성되었습니다.

## 2. 시스템 아키텍처
### 파이프라인 단계
1. **cron trigger** – 매주 평일 08:00 UTC에 `agentTurn` 작업 실행  
2. **repo pull** – `gh repo clone` 또는 `git pull` 로 빌드‑log 레포지토리 최신화  
3. **context read** – `SOUL.md`, `USER.md`, `memory/` 디렉터리에서 최신 컨텍스트 수집  
4. **topic selection** – 최근 며칠간 실제 발생한 사건 중 하나를 주제로 선택  
5. **markdown generation** – 날짜 기반 `.md` 파일 및 `README.md` 업데이트, 커밋·푸시  
6. **dev.to draft** – 초안 생성 → API 호출 → 게시  
7. **Telegram notification** – 제목·URL·한 줄 요약 전송  

### 데이터 저장소
- **SOUL.md / USER.md** – 에이전트의 장기·단기 메모 저장소  
- **memory/** – 최신 실행 로그, 빌드 결과 등 임시 파일  

### 외부 서비스 연동 포인트
| 서비스 | 연동 목적 | 주요 API |
|--------|-----------|----------|
| GitHub | 일기·빌드 로그 저장·버전 관리 | GitHub REST API ([docs](https://docs.github.com/en/rest)) |
| dev.to | 블로그 초안·게시 | dev.to API ([docs](https://developers.dev.to/api)) |
| Telegram Bot | 알림 전송 | Bot API (`sendMessage`) ([docs](https://core.telegram.org/bots/api#sendmessage)) |

## 3. 설정 및 크론 스케줄링
### OpenClaw 설치 및 초기 설정
1. OpenClaw 바이너리를 다운로드하고 실행 권한을 부여합니다.  
2. `config.yaml` 파일에 GitHub 레포지토리 URL, dev.to 토큰, Telegram Bot 토큰을 입력합니다.  
3. 초기 `agentTurn` 페이로드를 정의합니다.

### cron job 정의
```yaml
# OpenClaw cron schedule (UTC)
schedule: "0 8 * * 1-5"   # 매주 평일 08:00 UTC
task: agentTurn
payload:
  - pull_repo: true
  - read_context: true
  - select_topic: true
  - write_entry: true
  - commit_push: true
  - draft_devto: true
  - telegram_notify: true
```
*위 예시는 OpenClaw 설정 파일 형식이며, 실제 배포 환경에 맞게 경로와 옵션을 조정합니다.*

### 로컬·클라우드 환경 팁
- **Docker**: OpenClaw을 컨테이너화하고 `cron` 대신 `supercronic` 사용 가능.  
- **Kubernetes**: `CronJob` 리소스로 정의하면 클러스터 수준 스케줄링이 가능.  

## 4. OpenClaw API 연동
### `agentTurn` 호출 방식
OpenClaw은 내부 HTTP 엔드포인트 `/api/agentTurn`을 제공하며, POST 요청에 JSON 페이로드를 전달합니다.

```json
{
  "repo": "github.com/yourorg/build-log",
  "context_files": ["SOUL.md", "USER.md", "memory/"],
  "topic_strategy": "recent_events",
  "output": {
    "markdown_path": "diary/2024-03-27.md",
    "devto_publish": true,
    "telegram_notify": true
  }
}
```

### 인증
- **Bearer Token** 방식: `Authorization: Bearer <OPENCLAW_TOKEN>` 헤더 사용.  
- 토큰은 `config.yaml`에 저장된 비밀값을 사용합니다.

### 오류 처리 및 재시도
1. **HTTP 5xx** 응답 → 5분 간격으로 최대 3회 재시도.  
2. **GitHub/Dev.to/Telegram** 별도 오류 로그를 `logs/agentTurn.log`에 기록.  
3. 재시도 실패 시 **알림**을 Telegram으로 전송하고, 담당자(Julio)에게 이메일(또는 Slack)으로 에스컬레이션합니다.

## 5. GitHub 자동 커밋 프로세스
### 레포지토리 클론·업데이트 스크립트
```sh
gh repo clone yourorg/build-log repo
cd repo
git pull origin main
```

### 마크다운 일기 파일 생성 규칙
- 파일명: `YYYY-MM-DD.md` (예: `2024-03-27.md`)  
- 파일 헤더:  
  ```markdown
  # 일기 - 2024-03-27
  ```  
- 내용: 선택된 주제와 최신 메모리 파일에서 추출한 로그를 포함.

### 커밋 메시지 표준
```
feat(diary): add entry for 2024-03-27
```
*`feat(diary)` 프리픽스는 Conventional Commits 규칙을 따릅니다.*

### 자동 푸시
```sh
git add .
git commit -m "feat(diary): add entry for $(date +%F)"
git push origin main
```

## 6. dev.to 자동 포스팅
### 초안 생성 → API 호출 → 게시 흐름
1. **초안**: Markdown 파일을 `devto_draft.md` 로 복사.  
2. **API 호출**: `POST https://dev.to/api/articles` 에 JSON 바디와 `api-key` 헤더 전송.  
3. **응답**: `id`와 `url`을 받아 최종 게시(`PUT /api/articles/{id}/publish`).

### 인증 토큰 관리
- 토큰은 **환경 변수** `DEVTO_API_KEY` 로 설정하고, OpenClaw 설정 파일에 참조합니다.  
- 토큰은 최소 권한(`write_articles`)만 부여합니다.

### 메타데이터 자동 지정
```json
{
  "title": "자동 일기 - 2024-03-27",
  "published": false,
  "tags": ["automation", "build-log", "diary"],
  "canonical_url": "https://github.com/yourorg/build-log/blob/main/diary/2024-03-27.md"
}
```

## 7. Telegram 알림 자동 전송
### Bot 생성 및 토큰 확보
1. Telegram에서 **@BotFather**와 대화 → `/newbot` 명령 실행.  
2. Bot 이름·사용자명 지정 후 **Bot Token** 발급.  

### 알림 메시지 포맷
```
📝 새 일기 게시: 자동 일기 - 2024-03-27
🔗 https://dev.to/yourusername/auto-diary-2024-03-27
📄 한 줄 요약: 오늘 빌드 로그와 주요 변경 사항 정리
```

### 전송 스크립트 (inline)
```sh
curl -s -X POST "https://api.telegram.org/bot${TELEGRAM_TOKEN}/sendMessage" \
  -d chat_id=${CHAT_ID} \
  -d text="${MESSAGE}"
```

### 실패 시 재시도 및 로그
- **HTTP 429**(rate limit) → 10초 대기 후 재시도 (최대 3회).  
- 모든 시도 결과를 `logs/telegram.log`에 기록하고, 최종 실패 시 OpenClaw이 담당자에게 알림을 보냅니다.

## 8. 에이전트가 수행 가능한 작업 vs. 제한 사항
| 단계 | 셸 스크립트/자동화 가능 | 인간 판단 필요 |
|------|------------------------|----------------|
| 1 Pull repo | ✅ | ❌ |
| 2 Read context | ✅ | ❌ |
| 3 Topic selection | ❌ (복합 판단 필요) | ✅ |
| 4 Write entry | ✅ (템플릿 기반) | ❌ |
| 5 Commit & push | ✅ | ❌ |
| 6 Draft dev.to | ✅ (API 호출) | ❌ |
| 7 Telegram notify | ✅ | ❌ |

- **제한 사항**: 현재 LLM 기반 에이전트는 **주제 선정**과 **콘텐츠 생성** 단계에서 인간 수준의 이해와 창의성이 요구됩니다. 따라서 이 단계는 **수동 검토** 혹은 **보조 스크립트**와 결합해 운영합니다.

## 9. 예시 출력 및 검증 절차
### 자동 생성된 일기·포스트 예시
```
# 일기 - 2024-03-27

오늘은 CI 파이프라인에서 새로운 테스트 스위트를 추가했습니다.
- 테스트 커버리지: 78% → 85%
- 주요 변경점: auth 모듈 리팩터링
...
```
dev.to 초안 URL: `https://dev.to/yourusername/auto-diary-2024-03-27`

### 인간 감독 흐름
1. **Julio**가 Telegram 알림을 수신 → 제목·요약 검토.  
2. 이상 발견 시 **삭제** 명령(`DELETE /api/articles/{id}`)을 실행하거나 GitHub 커밋을 `git revert` 로 되돌림.  
3. 모든 검토 로그는 `audit/` 디렉터리에 보관.

## 10. 실패 모드와 완화 방안
| 실패 유형 | 증상 | 완화·대응 방안 |
|-----------|------|----------------|
| 환각된 컨텍스트 | 허위 내용 삽입 | 실제 메모리 파일 검증 후 내용 반영, 외삽 방지 로직 추가 |
| 시들한 주제 | 의미 없는 채우기 글 | 주제 부재 시 게시 건너뛰고 로그에 `no_topic` 기록 |
| API 오류 | GitHub/Dev.to/Telegram 전송 실패 | 작업 로그 남기고 재시도, 담당자 알림, 수동 복구 절차 마련 |
| 톤 드리프트 | 글 스타일 변질 | 정기적 인간 리뷰 및 스타일 가이드 적용, LLM 프롬프트 재조정 |

## 11. 인간 감독 및 거버넌스
- **검토 주기**: 매일 Telegram 알림 도착 후 30분 이내 검토.  
- **승인 권한**: `Julio`(Telegram 운영자)와 `repo_admin`(GitHub)에게 삭제·수정 권한 부여.  
- **고위험 콘텐츠**(법적·정책적) 발생 시 **추가 승인 단계**(예: 팀 리더) 적용.  

## 12. 자동화 가치 평가
- **일관성**: 주 1회 정기적인 빌드 로그 공개로 독자 기대 유지.  
- **효율성**: 개발자(Julio)가 문서화에 할당하는 시간 ≈ 10 시간/주 → 자동화로 0 시간으로 감소.  
- **분리된 책임**: 코드·빌드와 문서·홍보가 명확히 구분돼 운영 리스크 감소.

## 13. 확장 및 커스터마이징 가이드
### 새로운 출력 채널 추가
- **Twitter**: `POST https://api.twitter.com/2/tweets` 와 Bot Token 사용.  
- **Slack**: Incoming Webhook URL 로 JSON payload 전송.  

### 커스텀 주제 선정 로직
- 외부 RSS 피드·Jira 티켓을 스캔하고, 키워드 매칭 점수 기반으로 주제 후보 생성.  
- 스크립트 예시: `python select_topic.py --source rss --threshold 0.7`

### 스케줄링 빈도 조정
- `cron` 표현식 수정 (`0 8 * * *` → 매일 08:00) 혹은 **다중 에이전트** 운영 시 `agentTurn-1`, `agentTurn-2` 로 분산.
