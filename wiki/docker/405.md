## 개요
이 문서는 **Docker**와 **ZFS**를 결합해 홈랩 서버에 신뢰성 높고 효율적인 스토리지를 구축하고자 하는 개발자·시스템 관리자·홈랩 애호가를 대상으로 합니다.  
ZFS가 제공하는 **데이터 중복 제거, 압축, 스냅샷** 등 고급 기능을 Docker 볼륨과 연계하면, 컨테이너 기반 워크로드를 위한 **확장 가능하고 복구 가능한** 스토리지 인프라를 손쉽게 구현할 수 있습니다 ([Docker + ZFS 가이드](https://euno.news/posts/ko/docker-zfs-the-perfect-home-lab-storage-setup-af4849)).

## 전제 조건
| 항목 | 요구 사항 |
|------|-----------|
| 운영체제 | Ubuntu, FreeBSD, macOS 등 ZFS를 공식 지원하는 OS |
| 디스크 | 최소 2~3개의 물리 디스크 (HDD 또는 SSD) |
| Docker | `docker` 패키지가 설치·실행 중 |
| 기본 지식 | Linux CLI 기본 명령, Docker 기본 개념 |
| 기타 | 루트 권한(또는 `sudo` 사용 가능) |

## ZFS 설치 및 초기 설정
### 운영체제별 ZFS 패키지 설치
- **Ubuntu**: `sudo apt-get update && sudo apt-get install zfsutils-linux`  
- 기타 OS: 공식 ZFS 문서(예: FreeBSD Handbook) 참고 ([Docker + ZFS 가이드](https://euno.news/posts/ko/docker-zfs-the-perfect-home-lab-storage-setup-af4849)).

### 디스크 확인 및 식별
- 디스크 목록 확인: `lsblk`  
- 예시 디스크: `/dev/sdb`, `/dev/sdc`, `/dev/sdd`

## ZFS 풀 생성 실습
### 풀 생성 명령 예시
```text
sudo zpool create -f -o ashift=12 -o autoreplace=on tank raidz1 /dev/sdb /dev/sdc /dev/sdd
```
- `tank` : 풀 이름  
- `raidz1` : 단일 패리티 RAID‑Z (데이터 보호)  
- `ashift=12` : 4 KB 정렬 (현대 디스크에 권장)  
- `autoreplace=on` : 디스크 장애 시 자동 교체 활성화  

### 풀 상태 확인
`sudo zpool status`

### 설계 고려사항
- **풀 이름**은 전체 스토리지 정책을 반영하도록 직관적으로 지정  
- **RAID‑Z 레벨**은 디스크 수와 복구 목표에 따라 `raidz1`, `raidz2`, `mirror` 등 선택  

## 데이터셋 설계 및 구성
### 데이터셋 개념
ZFS 데이터셋은 풀 내 논리적 컨테이너이며, 각각 **Docker 볼륨**, **백업**, **공유 파일** 용도로 구분해 관리합니다.

### 데이터셋 생성 예시
- Docker 전용: `sudo zfs create tank/docker`  
- 백업 전용: `sudo zfs create tank/backups`  
- 공유 파일 전용: `sudo zfs create tank/shared`

### 주요 속성 튜닝
- **압축**: `sudo zfs set compression=lz4 tank/docker`  
- **중복 제거**(필요 시): `sudo zfs set dedup=on tank/docker`  
- **마운트 포인트** 확인: `sudo zfs get mountpoint tank/docker`

## Docker와 ZFS 볼륨 통합
### Docker 로컬 볼륨 드라이버에 ZFS 옵션 적용
```text
docker volume create \
  --driver local \
  --opt type=zfs \
  --opt device=tank/docker \
  --name docker-vol
```
- `--driver local` : 기본 로컬 볼륨 드라이버 사용  
- `--opt type=zfs` : ZFS 데이터셋 기반 지정  
- `--opt device=tank/docker` : 사용할 ZFS 데이터셋 지정  

### 볼륨 확인
`docker volume ls` → `docker-vol`이 목록에 표시됩니다.

### 컨테이너에 마운트 예시
```text
docker run -d --name web \
  -v docker-vol:/var/www/html \
  nginx:alpine
```
컨테이너 내부 `/var/www/html` 경로가 ZFS 기반 볼륨에 연결됩니다.

## 자동 스냅샷 설정
### 스냅샷 기본 원리
ZFS 스냅샷은 특정 시점의 데이터셋 상태를 **읽기 전용**으로 복제합니다. 스냅샷은 매우 빠르게 생성되며, 차등 저장 방식으로 디스크 사용량을 최소화합니다.

### 일일 스냅샷 스크립트 예시
```text
sudo zfs snapshot -r tank/docker@$(date +%Y-%m-%d)
```
위 명령은 `tank/docker`와 그 하위 데이터셋을 현재 날짜(`YYYY-MM-DD`) 태그와 함께 스냅샷합니다.

### cron을 이용한 자동화
루트 crontab(`sudo crontab -e`)에 다음 라인 추가:
```
0 0 * * * /sbin/zfs snapshot -r tank/docker@$(date +\%Y-\%m-\%d)
```
매일 자정에 스냅샷이 자동 생성됩니다 ([Docker + ZFS 가이드](https://euno.news/posts/ko/docker-zfs-the-perfect-home-lab-storage-setup-af4849)).

## 백업 및 복구 전략
### 스냅샷 기반 원격 백업 흐름
1. **스냅샷 생성** → `zfs snapshot -r tank/docker@daily`  
2. **전송**: `zfs send -R tank/docker@daily | ssh user@remote "zfs receive -F remote/tank/docker"`  
   - `-R` 옵션은 전체 계층을 포함해 전송  
3. **복구**: 원격에서 `zfs rollback tank/docker@daily` 실행

### rsync·클라우드 연동
- 스냅샷 마운트 지점을 `rsync -a /tank/docker/ user@remote:/backup/docker/` 로 복제  
- 클라우드 스토리지(AWS S3, Backblaze B2 등)와 연계하려면 `rclone` 사용

### 복구 테스트 체크리스트
- 최신 스냅샷 존재 여부 확인 (`zfs list -t snapshot`)  
- `zfs rollback` 실행 전 **데이터 손실 위험** 검토  
- 복구 후 컨테이너 재시작 및 로그 확인  

## 성능 최적화 및 모니터링
### 캐시 설정 팁
- **ARC**(Adaptive Replacement Cache)는 메모리 기반 캐시이며 기본 활성화  
- **L2ARC**(디스크 기반 캐시) 필요 시 SSD를 별도 디스크로 지정: `sudo zpool add tank cache /dev/sdx`

### 실시간 모니터링 명령
- 풀 I/O: `zpool iostat -v 1`  
- 데이터셋 속성: `zfs get all tank/docker`  
- Docker 컨테이너 I/O: `docker stats`  

### Docker I/O 튜닝 포인트
- 볼륨 마운트 옵션에 `noatime` 적용 (`sudo zfs set atime=off tank/docker`)  
- 컨테이너 실행 시 `--storage-opt size=...` 로 제한 설정 가능  

## 장애 대응 및 트러블슈팅
### 일반 오류와 원인
| 오류 | 원인 | 해결 방법 |
|------|------|----------|
| `zpool status: DEGRADED` | 디스크 손상 또는 탈락 | `zpool replace tank /dev/sdx` 후 `zpool scrub tank` |
| `zfs: dataset is busy` | 데이터셋이 마운트된 상태 | 해당 마운트를 해제(`zfs unmount`) 후 작업 수행 |
| Docker 볼륨 마운트 실패 | ZFS 데이터셋 권한 부족 | `sudo zfs set mountpoint=/var/lib/docker/volumes/tank/docker tank/docker` 및 `chmod` 조정 |

### 풀 정밀 검사
`sudo zpool scrub tank` → 진행 상황은 `zpool status`에서 확인

### Docker·ZFS 연동 시 권한 문제
- ZFS 데이터셋 소유자를 `docker` 그룹에 맞추기: `sudo chown -R 1000:1000 /tank/docker` (Docker 기본 UID/GID)  
- SELinux/AppArmor 정책이 적용된 경우, 해당 프로파일에 `zfs` 접근 허용 규칙 추가 필요  

## 보안 고려사항
- **데이터셋 ACL**: `sudo zfs set acltype=posixacl tank/docker` 로 POSIX ACL 활성화  
- **볼륨 접근 제어**: Docker 네임스페이스와 `--read-only` 옵션 사용 가능  
- **ZFS 암호화**(필요 시): `sudo zfs create -o encryption=on -o keyformat=passphrase tank/secure` (키 관리는 별도 문서 필요)  

## 마무리 및 향후 확장 방향
- **핵심 체크포인트**  
  1. ZFS 풀(`tank`)을 RAID‑Z1로 생성하고 상태 확인  
  2. Docker 전용 데이터셋(`tank/docker`)을 만들고 압축·중복 제거 설정  
  3. `docker volume create` 로 ZFS 기반 볼륨을 연결  
  4. `cron` 기반 일일 스냅샷 자동화 구현  
  5. `zfs send/receive` 로 원격 백업 파이프라인 구축  

- **추가 기능**  
  - **ZFS 레플리케이션**: `zfs send -R` 과 `zfs receive` 로 다중 사이트 복제  
  - **클러스터링**: `zfs` 자체는 클러스터링을 제공하지 않지만, `glusterfs`·`Ceph`와 연계 가능
