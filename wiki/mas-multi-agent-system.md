## 개요
다중 에이전트 시스템(Multi‑Agent System, **MAS**)은 **여러 인공지능(AI) 에이전트가 협력·조정하여** 사용자나 다른 시스템을 대신해 복합적인 작업을 수행하도록 설계된 시스템이다. 각 에이전트는 자체적인 속성과 자율성을 가지며, 전체 시스템은 **공통 목표**를 달성하기 위해 에이전트 간의 **커뮤니케이션·협업**을 활용한다.

## 핵심 구성 요소
| 요소 | 설명 |
|------|------|
| **에이전트** | LLM 기반 AI 에이전트로, 자연어 이해·생성, 도구 호출, 계획 수립 등을 수행한다. (출처: IBM) |
| **지식·메모리** | 에이전트는 외부 데이터, API, 웹 검색 등 도구를 활용해 정보를 획득하고, 메모리를 업데이트한다. |
| **통신 프로토콜** | 에이전트 간에 메시지를 주고받으며 목표·계획·결과를 공유한다. (Claude Code의 “agent teams” 참고) |
| **조정 메커니즘** | 중앙 집중식·분산식·계층형·홀로닉·연합·팀 등 다양한 아키텍처가 존재한다. (IBM) |

## 아키텍처 유형
1. **중앙 집중식 네트워크**  
   - 중앙 서버가 전역 지식 베이스와 에이전트 연결을 관리한다.  
   - 장점: 통신이 쉽고 지식이 일관됨.  
   - 단점: 중앙 서버 장애 시 전체 시스템이 중단될 위험이 있다. (IBM)

2. **분산형 네트워크**  
   - 에이전트가 인접 에이전트와 직접 정보를 교환한다.  
   - 장점: 견고하고 모듈성이 높으며 단일 장애점이 없음.  
   - 단점: 협업을 위한 행동 조정이 복잡할 수 있다. (IBM)

3. **계층형 구조**  
   - 트리 형태로 상위·하위 에이전트가 존재한다.  
   - 상위 에이전트가 의사결정 권한을 갖고, 하위 에이전트는 구체 작업을 수행한다. (IBM)

4. **홀로닉 구조**  
   - 에이전트가 “홀론” 단위로 그룹화되어, 하나의 전체와 여러 하위 에이전트가 동시에 존재한다. (IBM)

5. **연합·팀 구조**  
   - 에이전트가 일시적으로 연합하거나 팀을 이루어 특정 목표를 달성한다. (IBM)

## 에이전트 협업 방식
### 1. Agent Teams (Claude Code)
- **팀 리드**가 작업을 분배하고 결과를 종합한다.  
- **팀원**은 독립적인 컨텍스트 창에서 작업하며 서로 직접 메시지를 주고받을 수 있다.  
- **사용 사례**: 연구·리뷰, 새로운 모듈 설계, 경쟁 가설 디버깅, 프론트·백엔드·테스트 동시 변경. (Claude Docs)  
- **비용**: 각 팀원이 별도 Claude 인스턴스를 사용하므로 토큰 사용량이 증가한다. (Claude Docs)

### 2. Subagents
- 메인 세션 내부에서 실행되는 보조 에이전트.  
- 결과는 메인 에이전트에만 보고되며, 팀원 간 직접 통신은 불가능하다. (Claude Docs)

| 구분 | Context | Communication | Coordination |
|------|---------|---------------|--------------|
| **Agent Teams** | 각 팀원 별도 컨텍스트 | 팀원 ↔ 팀원 직접 | 공유 작업 리스트, 자체 조정 |
| **Subagents** | 메인 세션 내 | 결과 → 메인만 | 메인 에이전트가 전체 관리 |

## 주요 행동 패턴
| 행동 | 설명 |
|------|------|
| **플로킹 (Flocking)** | 에이전트가 목표를 공유하고, 충돌 회피·속도 정렬·집합성을 유지한다. (IBM) |
| **스워밍 (Swarming)** | 분산 제어를 통한 자기 조직화·집계가 발생한다. 대규모 에이전트가 협업해 복잡 문제를 해결한다. (IBM) |

## 활용 사례
| 분야 | 구체 예시 |
|------|----------|
| **운송** | 철도 스케줄링, 트럭 배정, 항구 선박 흐름 관리 |
| **의료·공중보건** | 유전자 분석 기반 질병 예측, 전염병 확산 시뮬레이션 |
| **공급망 관리** | 가상 에이전트 간 협상·재고 최적화 |
| **방위·보안** | 해상 공격 시뮬레이션, DDoS 탐지·대응 |
| **소프트웨어 개발** | 병렬 코드 리뷰, 기능 설계·구현 협업 (Claude Code “agent teams”) |

## 장점
- **유연성**: 에이전트를 추가·제거·조정하여 다양한 환경에 대응 가능.
- **확장성**: 다수 에이전트가 협력해 복잡 문제를 처리, 단일 에이전트보다 높은 성능 기대.
- **도메인 전문화**: 각 에이전트가 특정 분야에 특화되어 전체 시스템의 전문성을 향상.
- **성능 향상**: 다중 계획·학습·피드백을 통해 더 빠른 문제 해결이 가능.

## 과제 및 한계
| 과제 | 설명 |
|------|------|
| **에이전트 오작동** | 동일 기반 모델 사용 시 공유된 취약점이 전체 시스템에 영향을 미칠 수 있다. |
| **조정 복잡성** | 협업·협상이 필요한 에이전트를 설계·관리하는 것이 어려움. |
| **예측 불가능한 행동** | 자율적인 분산 에이전트가 충돌하거나 비정상적인 행동을 보일 수 있다. |
| **토큰·리소스 비용** | 특히 Agent Teams는 각 인스턴스당 토큰 사용량이 크게 증가한다. (Claude Docs) |

## 베스트 프랙티스
1. **작업을 독립적으로 나누기** – 팀원 간 최소한의 의존성을 유지하도록 설계한다. (Claude Docs)
2. **적절한 규모 선택** – 작은 순차 작업은 단일 세션이나 Subagents가 효율적이며, 복잡·병렬 작업에만 Agent Teams 사용. (Claude Docs)
3. **플랜 승인 활용** – 위험하거나 복잡한 변경은 “plan approval” 모드로 검증 후 실행한다. (Claude Docs)
4. **모니터링 및 스톱워치** – 팀 리드가 진행 상황을 지속적으로 확인하고 필요 시 팀원을 종료한다. (Claude Docs)
5. **환경에 맞는 아키텍처 선택** – 중앙 집중식이 필요한 경우는 관리 편의성을, 분산형이 필요한 경우는 견고성을 우선한다. (IBM)

## 설정 예시 (Claude Code – Agent Teams)
```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  },
  "teammateMode": "in-process"
}
```
> 위 설정은 **Agent Teams** 기능을 활성화하고, 모든 팀원을 현재 터미널 프로세스 내에서 실행하도록 지정한다. (Claude Docs)
