## 소개
- **문서 목적**: GitHub Actions 2026 보안 로드맵을 이해하고, 조직의 CI/CD 파이프라인에 적용하기 위한 실무 가이드를 제공한다.  
- **대상 독자**: DevSecOps 엔지니어, 플랫폼 운영팀, 보안 담당자, 개발 팀 리더 등 CI/CD 보안에 관심이 있는 모든 기술 관계자.  
- **GitHub Actions 2026 보안 로드맵 개요**: GitHub은 소프트웨어 공급망 공격을 방지하기 위해 *에코시스템 → 공격 표면 → 인프라스트럭처* 3계층 보안 모델을 도입한다. 이는 보안 기본값을 기본으로 하고, 정책 제어와 실시간 관측성을 통해 엔드‑투‑엔드 방어를 구현한다.  
- **핵심 용어 정의**  
  - **CI/CD**: 지속적 통합·배포 파이프라인.  
  - **소프트웨어 공급망**: 코드, 의존성, 빌드·배포 과정 전체를 아우르는 생태계.  
  - **정책 제어 (Policy Controls)**: 워크플로 실행 전·후에 적용되는 자동화된 보안 규칙.  
  - **관측성 (Observability)**: 실행 로그, 메트릭, 알림 등을 통한 실시간 가시성 확보.  

## 현재 위협 상황
- **최근 공급망 공격 사례**  
  - `tj-actions/changed-files`·`Nx`·`trivy-action` 등 GitHub Actions 자체를 목표로 한 공격이 연속적으로 보고되었다[GitHub Blog](https://github.blog/news-insights/product-news/whats-coming-to-our-github-actions-2026-security-roadmap/).  
- **공격 패턴**  
  1. **취약점 이용**: 의존성에 존재하는 취약점으로 비신뢰 코드 실행.  
  2. **악성 워크플로**: 관측성·제어가 부족한 워크플로가 무단 실행.  
  3. **의존성 전파**: 취약한 Action이 트랜시티브 의존성을 통해 다수 레포지토리에 퍼짐.  
  4. **과다 권한 크리덴셜**: 제한 없는 네트워크 접근으로 크리덴셜이 탈취.  
- **기존 보안 격차**  
  - **비결정적 의존성**: 태그·브랜치 등 가변 참조 사용으로 실행 내용이 고정되지 않음.  
  - **투명성 부족**: Action 퍼블리싱 과정에서 서명·검증이 미비.  
  - **정책 부재**: 워크플로에 대한 사전 검증·제어가 제한적.  

## 로드맵 전체 구조
- **3계층 보안 모델**  
  1. **Ecosystem (에코시스템)** – 결정론적(immutable) 의존성, 서명·투명성 로그 강화.  
  2. **Attack Surface (공격 표면)** – Secure Defaults, 정책 기반 접근 제어(PBAC), 스코프된 크리덴셜.  
  3. **Infrastructure (인프라스트럭처)** – 실시간 관측성, 네트워크 경계 격리.  
- **보안 기본값 → 정책 제어 → 관측성** 흐름은 “보안이 기본이 되도록” 설계되었으며, 각 단계는 이전 단계의 결과를 기반으로 강화된다.  
- **적용 범위와 기대 효과**: 모든 퍼블릭·프라이빗 레포지토리와 자체 호스팅 러너에 적용되어, 공급망 공격의 초기 진입점을 차단하고, 침해 시 빠른 탐지·대응이 가능하도록 한다.  

## 에코시스템 보안 강화
- **결정론적 의존성 도입**  
  - 워크플로에서 **커밋 SHA**를 사용하도록 권장하고, 태그·브랜치 대신 불변 해시를 기본값으로 설정한다. 이는 실행 시점에 의존성이 변하지 않도록 보장한다[GitHub Blog](https://github.blog/news-insights/product-news/whats-coming-to-our-github-actions-2026-security-roadmap/).  
- **Action 퍼블리싱 보안**  
  - Action 배포 시 **디지털 서명**과 **투명성 로그**를 제공해, 사용자는 서명 검증을 통해 신뢰성을 확인할 수 있다.  
- **트랜시티브 의존성 가시화**  
  - 새로운 도구가 도입되어, Action이 의존하는 모든 하위 Action·패키지를 자동으로 탐색·표시한다. 현재 구체적인 도구명은 공개되지 않았으며, 추가 조사가 필요합니다.  

## 공격 표면 감소 정책
- **보안 기본값 (Secure Defaults)**  
  - 기본적으로 **읽기 전용 토큰**이 제공되며, 네트워크 아웃바운드가 차단된 환경에서 실행된다.  
- **정책 기반 접근 제어 (PBAC)**  
  - 조직 수준에서 **워크플로 검증 규칙**(예: 허용된 Action 목록, SHA 검증)과 **정책 위반 시 자동 차단** 옵션을 설정할 수 있다.  
- **스코프된 크리덴셜 관리**  
  - 크리덴셜은 최소 권한 원칙에 따라 **스코프 제한**되어 제공되며, 필요 시 **시간 제한** 토큰을 사용할 수 있다.  
- **네트워크 접근 제한**  
  - 러너에 **아웃바운드 네트워크 경계**를 정의해, 외부 서비스와의 직접 연결을 차단하고, 허용된 엔드포인트만 접근하도록 한다.  

## 인프라스트럭처 관측성
- **실시간 실행 로그·메트릭**  
  - 워크플로 실행 시 **스트리밍 로그**와 **성능 메트릭**이 GitHub UI와 API를 통해 제공된다.  
- **러너 네트워크 경계 격리**  
  - 자체 호스팅 러너는 **가상 네트워크** 내에서 격리되며, 정책에 따라 **네트워크 세그멘테이션**이 적용된다.  
- **AI/ML 기반 이상 징후 탐지**  
  - GitHub은 **머신러닝 모델**을 활용해 비정상적인 실행 패턴을 실시간으로 감지하고, 알림을 발송한다.  
- **감사·포렌식 데이터 보존**  
  - 로그와 메트릭은 **규정된 보존 기간** 동안 암호화 저장되며, 접근 제어가 적용된다.

## 마이그레이션 가이드
- **영향 분석 체크리스트**  
  1. 워크플로에 사용된 태그·브랜치 확인  
  2. 현재 크리덴셜 스코프 파악  
  3. 네트워크 아웃바운드 요구사항 검토  
- **전환 전략**  
  1. **파일 해시 적용**: 기존 Action 호출을 SHA 기반으로 교체.  
  2. **정책 적용**: 조직 레벨 정책 파일(`.github/policies.yml`)을 추가하고, CI/CD 파이프라인에 적용.  
  3. **관측성 연동**: 새로운 로그 스트리밍 API를 활성화하고, 대시보드에 연결.  
- **자동화 도구·스크립트**: GitHub에서 제공하는 **migration CLI**가 베타 단계에 있으며, 상세 명령어는 추후 공개될 예정(추가 조사가 필요합니다).  
- **롤백·복구 절차**: 정책 적용 전 스냅샷을 저장하고, 문제가 발생하면 기존 태그·브랜치 방식으로 되돌릴 수 있다.  

## 베스트 프랙티스
- **최소 권한 원칙**: 크리덴셜은 필요한 최소 권한만 부여하고, 사용 후 즉시 폐기한다.  
- **의존성 관리 자동화**: Dependabot·Renovate와 같은 도구를 활용해 SHA 기반 업데이트를 자동화한다.  
- **정기적인 보안 정책 리뷰**: 조직 정책을 분기별로 검토·갱신하여 최신 위협에 대응한다.  
- **CI/CD 보안 교육**: 개발자와 운영자를 대상으로 워크플로 보안 원칙과 정책 적용 방법을 교육한다.  

## 모니터링 및 감사
- **주요 보안 지표(KPI)**  
  - SHA 기반 Action 사용 비율  
  - 정책 위반 감지 건수  
  - 실시간 알림 응답 시간  
- **대시보드 예시**: GitHub Security Insights에 통합된 **CI/CD 보안 대시보드**를 통해 위 KPI를 시각화한다.  
- **감사 로그 보관**: 로그는 최소 90일(조직 정책에 따라 연장 가능) 동안 보관되며, 접근은 **역할 기반 접근 제어(RBAC)**로 제한한다.  

## FAQ
**Q1. “Secure Defaults”가 구체적으로 무엇인가요?**  
A: 기본적으로 워크플로는 읽기 전용 토큰과 차단된 네트워크 환경에서 실행되며, 의도하지 않은 외부 호출을 방지한다.  

**Q2. 기존 Action을 그대로 사용할 수 있나요?**  
A: 기존 태그·브랜치 기반 Action은 여전히 동작하지만, 보안 권고에 따라 **SHA 기반**으로 전환하는 것이 권장된다.  

**Q3. 정책 위반 시 자동 차단 옵션은?**  
A: PBAC 정책 파일에 `enforce: true` 설정을 추가하면, 위반 워크플로는 실행 자체가 차단된다.  

**Q4. 비용 및 퍼포먼스 영향은?**  
A: 기본값 변경과 관측성 기능은 기존 요금제에 포함되며, 추가적인 네트워크 격리 옵션은 러너 유형에 따라 소폭 비용 변동이 있을 수 있다. 정확한 비용 모델은 GitHub Docs에 명시될 예정이며, 현재는 추가 조사가 필요합니다.
