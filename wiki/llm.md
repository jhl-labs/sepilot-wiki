## 1. 문서 개요
### 목적 및 대상 독자
본 가이드는 LLM(대형 언어 모델) 기반 서비스 운영 시 발생할 수 있는 **프롬프트 인젝션** 위협을 실시간으로 탐지하고 자동 차단하기 위한 프레임워크 설계·구현 방법을 제공한다.  
대상 독자는 보안 엔지니어, 플랫폼 운영팀, DevOps 엔지니어, AI 제품 관리자 등 LLM을 서비스에 통합하고 보안·관측성을 강화하고자 하는 모든 기술 담당자이다.

### 프레임워크 정의와 기대 효과
- **프레임워크**: 로그·메트릭 수집 → 전처리 → 탐지 엔진 → 차단·완화 → 관측성 연동의 일련 흐름을 표준화한 시스템이다.  
- **기대 효과**  
  - 인젝션 시도에 대한 **실시간 탐지**와 **자동 차단** 구현  
  - 보안·관측성 태그를 활용한 **통합 모니터링**  
  - 정책 기반 **자동화**와 **감사 추적**을 통한 운영 효율성 향상  

### 기존 방어 가이드와 차별점
기존 가이드는 주로 **입력 정제**와 **프롬프트 설계** 단계에서 방어를 권고한다. 본 프레임워크는  
1. **관측 기반 탐지**(로그·메트릭)  
2. **다중 탐지 기법**(패턴, 이상 행동, 보조 LLM)  
3. **자동 차단 및 정책 업데이트**  
를 추가하여 방어·탐지를 **통합**하고, 실시간 대응을 가능하게 한다.

## 2. 프롬프트 인젝션 위협 모델
### 인젝션 유형
| 유형 | 설명 |
|------|------|
| 프롬프트 변조 | 사용자가 제공한 프롬프트에 악의적인 명령어나 컨텍스트를 삽입 |
| 컨텍스트 탈취 | 이전 대화 내용이나 시스템 프롬프트를 탈취해 비정상적인 동작 유도 |
| 악성 명령 삽입 | 모델에게 실행 가능한 스크립트, 파일 경로, API 키 등을 전달 |

### 공격 흐름 및 공격 표면
1. **입력 단계** – 사용자/외부 시스템이 프롬프트를 전송  
2. **전달 단계** – API 게이트웨이·프록시를 거쳐 LLM 엔진에 전달  
3. **처리 단계** – LLM이 프롬프트를 해석·생성  
4. **응답 단계** – 결과가 반환되고, 후속 작업이 실행될 수 있음  

각 단계마다 **입력 검증**, **로그 기록**, **정책 적용**이 가능한 지점이 존재한다.

### 위험도 평가 기준
- **시도 빈도**: 동일 IP·계정에서 반복 시도 여부  
- **인젝션 복잡도**: 단순 문자열 삽입 vs. 복합 토큰 시퀀스  
- **잠재 영향**: 데이터 유출, 시스템 명령 실행, 서비스 중단 등  

위험도는 위 세 요소를 조합해 **고·중·저** 등급으로 분류한다.

## 3. 탐지 프레임워크 아키텍처
### 핵심 컴포넌트
- **수집기**: LLM 호출 로그·메트릭을 실시간 스트림으로 수집  
- **전처리**: 로그 정규화·필터링, 보안·관측성 태그 부착  
- **탐지 엔진**: 패턴 매칭, 이상 행동 분석, 보조 LLM 검증 수행  
- **차단 모듈**: 정책 기반 요청 차단·샌드박스 전송, 응답 변조  
- **관측성 연동**: 메트릭·알림을 외부 대시보드·알림 시스템에 전달  

### 데이터 흐름 다이어그램 (텍스트 설명)
1. 클라이언트 → **API 게이트웨이** (수집기) → 로그/메트릭 전송  
2. **전처리** 단계에서 로그를 표준 포맷으로 변환하고 `security:prompt_injection` 태그 부착  
3. **탐지 엔진**이 전처리된 데이터를 분석하고 스코어링  
4. 스코어가 임계값을 초과하면 **차단 모듈**이 정책에 따라 요청을 차단하거나 샌드박스로 전송  
5. 결과와 차단 로그는 **관측성 연동**을 통해 Grafana/Kibana 등 대시보드에 전송 및 알림 트리거  

### 배포 옵션
- **온프레미스**: 자체 데이터센터에 Docker Compose 혹은 쿠버네티스 클러스터로 배포  
- **클라우드**: 관리형 컨테이너 서비스(AKS/EKS/GKE)와 클라우드 로그 서비스 연동  
- **하이브리드**: 민감 데이터는 온프레미스, 비민감 메트릭은 클라우드에 전송

## 4. 실시간 로그·메트릭 수집
### LLM 호출 로그 포맷 표준화
| 필드 | 설명 |
|------|------|
| timestamp | 요청 발생 시각 |
| request_id | 고유 요청 식별자 |
| user_id / client_ip | 호출 주체 정보 |
| prompt | 실제 전달된 프롬프트 문자열 |
| model | 사용된 LLM 버전 |
| response_status | 성공/실패 여부 |
| security_tags | 보안·관측성 태그 (예: `security:prompt_injection`) |

### 보안·관측성 태그 정의
- `security:prompt_injection` – 인젝션 의심 로그에 부착  
- `security:blocked` – 차단된 요청에 부착  
- `observability:latency` – 응답 지연 메트릭에 부착  

### Logmera 등 관측성 도구와 연동 방법
1. **Logmera 에이전트**를 LLM API 서버에 설치하고 위 표준 포맷을 전송하도록 설정  
2. **Logmera 파이프라인**에서 `security:*` 태그를 필터링하여 별도 인덱스로 라우팅  
3. **대시보드**와 **알림**은 Logmera UI 혹은 외부 Grafana 플러그인을 통해 구성  

> 구체적인 연동 방법은 각 도구의 공식 문서를 참조해주세요.

## 5. 탐지 기법
### 정규식·패턴 매칭 기반 탐지
- 일반적인 명령어(`rm`, `curl`, `wget` 등)와 특수 토큰(`<<SYS>>`, `{{` 등) 패턴을 사전 정의  
- 매칭 시 **스코어**를 부여하고 임계값 초과 시 차단  

### 이상 행동 기반 탐지
- **시퀀스 이상**: 정상 프롬프트 길이·구조와 크게 벗어나는 경우 감지  
- **토큰 분포**: 특정 토큰(예: 시스템 프롬프트 토큰)의 비정상적 빈도 상승 탐지  

### LLM 자체 검증 모델 활용 (secondary LLM)
- 메인 LLM 호출 전, 경량 **보조 LLM**에 동일 프롬프트를 전달하고 **안전성 점수**를 반환받는다.  
- 보조 모델이 위험 점수를 높게 평가하면 메인 요청을 차단하거나 재검증한다.  

### 멀티‑모델 앙상블 및 스코어링 방식
- 패턴 매칭, 이상 행동, 보조 LLM 점수를 **가중 평균**하거나 **룰 기반**으로 결합  
- 최종 스코어가 정책 임계값을 초과하면 차단 모듈에 전달

## 6. 자동 차단 및 완화 전략
### 사전 정의 차단 정책
- **샌드박스**: 의심 요청을 격리된 환경에서 실행하고 결과를 검증  
- **입력 정제**: 위험 토큰을 마스킹하거나 삭제 후 재전송  
- **요청 거부**: 위험도가 높을 경우 HTTP 403/429 응답 반환  

### 동적 정책 업데이트 메커니즘
- 탐지 엔진이 새로운 인젝션 패턴을 학습하면 **정책 저장소**(예: ConfigMap) 를 자동 업데이트  
- 업데이트된 정책은 **핫 리로드**를 통해 차단 모듈에 즉시 반영  

### 차단 로그와 감사 추적
- 차단 시 `security:blocked` 태그와 함께 **차단 사유**, **정책 버전**, **처리 시간**을 로그에 기록  
- 감사 로그는 **읽기 전용 스토리지**에 보관하고, 규정 준수 검토 시 활용

## 7. 관측성 및 알림 연계
### 메트릭 정의
- `prompt_injection_detection_total` – 탐지된 인젝션 시도 총수  
- `prompt_injection_blocked_total` – 차단된 시도 총수  
- `prompt_injection_severity{level="high|medium|low"}` – 위험도 별 건수  
- `prompt_injection_latency_seconds` – 탐지·차단 처리 지연 시간  

### 대시보드 설계 예시
- **Grafana**: 위 메트릭을 시계열 그래프와 파이 차트로 시각화  
- **Kibana**: `security:prompt_injection` 태그가 포함된 로그를 필터링하여 상세 검색 가능  

### 알림 채널 설정 가이드
1. **Alertmanager**(Prometheus) 혹은 **Logmera 알림**에 웹훅을 등록  
2. 알림 조건: `prompt_injection_blocked_total`이 일정 시간 내에 급증하거나 `severity="high"` 발생 시  
3. 채널: Slack, PagerDuty, 이메일 등 조직에 맞는 도구와 연동  

> 알림 설정 시 과도한 노이즈를 방지하기 위해 **억제 규칙**을 적용하는 것이 권장됩니다.

## 8. 구현 예시
### 샘플 파이프라인 (Docker Compose / Kubernetes)
- **Docker Compose**: `collector`, `preprocessor`, `detector`, `blocker`, `observability` 컨테이너를 정의하고 네트워크로 연결  
- **Kubernetes**: 각 컴포넌트를 Deployment와 Service로 배포하고, ConfigMap/Secret을 통해 정책을 관리  

### 주요 오픈소스 라이브러리 및 SDK 소개
- **Fluent Bit / Fluentd** – 로그 수집 및 전송  
- **Prometheus Exporter** – 메트릭 노출  
- **OpenAI / HuggingFace Transformers** – 보조 LLM 호출 SDK  
- **OPA (Open Policy Agent)** – 정책 정의와 실시간 평가  

### CI/CD 파이프라인에 통합하는 방법
1. **코드 검증** 단계에서 Linter와 보안 스캔 수행  
2. **컨테이너 이미지 빌드** 후 자동 테스트(정상·위험 프롬프트 시뮬레이션) 실행  
3. **배포** 단계에서 Helm 차트 혹은 Docker Compose 파일에 최신 정책 ConfigMap을 포함

## 9. 운영 및 유지보수
### 정책 검토 주기와 업데이트 절차
- **주간**: 신규 탐지 패턴 및 차단 로그 검토  
- **월간**: 정책 버전 관리 및 위험도 재평가  
- **분기별**: 전체 프레임워크 성능 리뷰와 개선 계획 수립  

### 성능 튜닝 포인트
- **레이트 리밋**: 초당 요청 수 제한으로 탐지 엔진 부하 방지  
- **배치 처리**: 로그를 일정량씩 묶어 전처리·탐지 수행 시 처리량 향상  
- **캐시**: 보조 LLM 검증 결과를 단기 캐시하여 중복 호출 최소화  

### 로그 보관 및 GDPR/CCPA 준수 방안
- **로그 보관 기간**을 정책에 명시하고, 자동 삭제 스케줄링 적용  
- **민감 데이터 마스킹**: 사용자 식별 정보는 별도 암호화 혹은 마스킹 후 저장  
- **데이터 접근 제어**: 최소 권한 원칙에 따라 로그 조회 권한을 제한

## 10. 사고 대응 프로세스
### 단계별 흐름
1. **탐지** – 실시간 알림이 발생하면 담당자가 즉시 확인  
2. **알림** – Slack/PagerDuty 등에 상세 로그와 위험도 전달  
3. **조사** – 로그와 메트릭을 분석하여 인젝션 원인 및 영향을 파악  
4. **차단** – 필요 시 정책을 즉시 강화하거나 수동 차단 적용  
5. **복구** – 영향을 받은 시스템을 원상 복구하고, 사후 보고서 작성  

### 포렌식 로그 수집 및 분석 체크리스트
- 요청 ID, 타임스탬프, 원본 프롬프트, 전처리 결과, 탐지 스코어, 차단 사유  
- 네트워크 트래픽 캡처(필요 시)  
- 보조 LLM 검증 결과와 해당 모델 버전

## 11. 베스트 프랙티스 및 권고사항
### 프롬프트 설계 시 보안 원칙 적용
- **최소 권한 원칙**: 프롬프트에 필요한 최소 정보만 포함  
- **정적 템플릿 사용**: 사용자 입력을 변수 형태로 삽입하고, 템플릿 자체는 변조되지 않도록 관리  
- **입력 검증**: 길이, 문자 집합, 허용된 키워드 등을 사전 검증  

### 팀 내 협업 및 교육 가이드라인
- 정기적인 **보안 워크숍**을 통해 인젝션 사례와 대응 방법 공유  
- **코드 리뷰** 시 프롬프트 처리 로직에 대한 보안 체크리스트 적용  
- **문서화**: 정책, 탐지 규칙, 차단 로그 포맷을 중앙 위키에 유지  

### 외부 위협 인텔리전스와 연계 방안
- 공개된 **프롬프트 인젝션 IOCs**(Indicators of Compromise)를 정기적으로 수집  
- 위협 인텔리전스 피드와 탐지 엔진을 **자동 동기화**하도록 스크립트 구성

## 12. 향후 확장 로드맵
### 새로운 인젝션 기법 대응 전략
- **멀티턴 대화**에서 발생하는 컨텍스트 탈취 탐지 모델 추가  
- **코드 실행형 프롬프트**(예: Python REPL) 전용 샌드박스와 정밀 검증 도입  

### 자동 학습 기반 탐지 모델 고도화 계획
- 실시간 로그를 활용한 **온라인 학습** 파이프라인 구축  
- **피드백 루프**: 차단·허용 결과를 라벨링 데이터로 활용하여 모델 재학습  

### 커뮤니티 기여 및 오픈소스 생태계 연계
