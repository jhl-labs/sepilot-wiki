## 개요
- **문서 목적**: Traefik File Provider를 사용하면서 발생하는 반복적인 YAML 편집 작업을 경감하고, 가벼운 웹 UI인 **Traefik Route Manager (TRM)** 를 통해 라우트를 효율적으로 관리하는 방법을 제공한다.  
- **대상 독자**: 홈랩·NAS 운영자, 소규모 서비스 운영자, Traefik을 파일 기반으로 운영 중이며 라우트 관리 자동화를 원하는 개발자·운영자.  
- **핵심 가치**  
  - 데이터베이스·Redis 등 외부 의존성 없이 파일 기반 라우트 생성.  
  - 도메인당 하나의 `trm-{domain}.yml` 파일로 가독성·버전 관리 용이.  
  - 기존 Traefik 동적 구성 흐름과 완전 호환.  
  - AI 어시스턴트 연동을 위한 API·스킬 제공[[euno.news](https://euno.news/posts/ko/tired-of-hand-editing-traefik-yaml-this-little-too-245e37)].

## 기존 파일 기반 라우트 관리의 문제점
| 문제 | 상세 내용 |
|------|-----------|
| **수동 YAML 편집** | 새로운 서브도메인·백엔드 추가 시 매번 파일을 작성·수정해야 함. 오류 발생 위험이 높음. |
| **다중 라우트 파일** | 도메인·서브도메인마다 별도 파일이 쌓여 관리 복잡도 증가. |
| **HTTPS·리다이렉트 설정** | entrypoint, TLS, 리다이렉트 규칙을 일일이 수정해야 함. |
| **대규모 홈랩 부담** | 다수 서비스가 늘어나면 작은 유지보수 작업도 반복적으로 발생. |
| **버전·백업 관리 어려움** | 파일이 흩어져 있어 Git 등으로 일관된 버전 관리가 힘듦. |

## Traefik Route Manager 소개
- **경량 웹 UI**: Go 백엔드와 임베디드 React 프론트엔드로 구성된 단일 바이너리(~15 MB) 이미지[[GitHub](https://github.com/jae-jae/traefik-route-manager)].
- **데이터베이스‑프리 설계**: 파일 시스템에 `trm-{domain}.yml` 형태로 라우트를 저장, 외부 DB·Redis 불필요.
- **Traefik 호환**: 생성된 YAML은 표준 Traefik 동적 구성 파일이므로 기존 `file` 프로바이더 디렉터리에 그대로 마운트하면 자동 적용.
- **주요 기능**  
  - 도메인·백엔드 URL 입력 → 자동 YAML 생성.  
  - HTTPS 토글 및 HTTP→HTTPS 리다이렉트 옵션.  
  - AI 에이전트 연동을 위한 **Skill** 및 **Agents** 문서 제공[[GitHub/AGENTS.md](https://github.com/jae-jae/traefik-route-manager/blob/main/AGENTS.md)].  

## 시스템 아키텍처
- **구성 요소**  
  - **Go 백엔드**: 라우트 파일 생성·저장·감시 로직 담당.  
  - **React 프론트엔드**: 모바일·데스크톱 친화 UI 제공.  
  - **파일 시스템**: `CONFIG_DIR` 환경변수로 지정된 디렉터리에 `trm-*.yml` 저장.  
- **컨테이너 이미지**: `ghcr.io/jae-jae/traefik-route-manager:main` (Docker Hub 대신 GitHub Container Registry).  
- **인증**: `AUTH_TOKEN` 환경변수 기반 토큰 인증, HTTP 헤더 `Authorization: Bearer <token>` 사용.  
- **보안 경계**: UI와 API는 동일 토큰을 공유하므로, 외부 접근 차단(예: 방화벽, Traefik 자체 인증) 권고.

## 설치 및 배포 옵션
### Docker 단일 명령
```text
docker run -d \
  --name traefik-route-manager \
  -p 8892:8892 \
  -v /path/to/traefik/dynamic:/data \
  -e AUTH_TOKEN=your-secret-token \
  -e CONFIG_DIR=/data \
  ghcr.io/jae-jae/traefik-route-manager:main
```
- `-v` 옵션에 Traefik이 감시 중인 동적 구성 디렉터리를 마운트한다.  
- `AUTH_TOKEN`은 임의의 비밀 문자열이며, UI 로그인 및 API 호출 시 사용한다.

### Docker Compose 예시
```text
version: "3.8"
services:
  traefik-route-manager:
    image: ghcr.io/jae-jae/traefik-route-manager:main
    container_name: traefik-route-manager
    ports:
      - "8892:8892"
    volumes:
      - /path/to/traefik/dynamic:/data
    environment:
      AUTH_TOKEN: your-secret-token
      CONFIG_DIR: /data
```
- `docker compose up -d` 로 실행 후 `http://<host>:8892` 에 접속한다.

### 환경 변수 가이드
| 변수 | 설명 | 예시 |
|------|------|------|
| `AUTH_TOKEN` | UI·API 인증에 사용되는 공유 토큰 | `mySuperSecret123` |
| `CONFIG_DIR` | 라우트 파일이 저장될 디렉터리 (컨테이너 내부 경로) | `/data` |
| `HOST` (선택) | UI가 바인딩될 호스트 (기본 `0.0.0.0`) | `0.0.0.0` |

## 기본 사용 방법
1. **UI 접속**: 브라우저에서 `http://<host>:8892` 로 이동. 로그인 화면이 나오면 `AUTH_TOKEN` 값을 입력한다.  
2. **새 라우트 추가**  
   - **Domain**: 관리하고자 하는 도메인 (예: `plex.example.com`).  
   - **Backend URL**: 서비스가 노출되는 내부 주소 (예: `http://192.168.1.10:32400`).  
   - **HTTPS**: 토글을 켜면 `entryPoints: ["websecure"]` 와 TLS 설정이 자동 삽입.  
   - **Redirect**: HTTP→HTTPS 리다이렉트를 원하면 옵션 활성화.  
   - **저장** 버튼 클릭 → `trm-plex.example.com.yml` 파일이 `CONFIG_DIR`에 생성.  
3. **파일 확인**: 컨테이너 내부 혹은 호스트 마운트 디렉터리에서 생성된 YAML을 확인한다. 내용은 표준 Traefik 동적 라우트 형식과 동일.  

## 고급 기능
- **AI 에이전트 연동**  
  - `AGENTS.md`와 `SKILL.md`에 정의된 스킬을 활용해 자연어 명령으로 라우트를 추가·수정할 수 있다. 예: “Add a route for my Home Assistant at home.example.com”.  
  - 해당 스킬은 OpenAI‑compatible AI 모델에 로드하여 사용한다(구체적인 API 호출 방법은 `AGENTS.md` 참고).  
- **다중 엔트리포인트·TLS 커스터마이징**  
  - UI에서 기본 HTTPS 외에 추가 `entryPoints`(예: `web`, `websecure`)를 선택 가능.  
  - 필요 시 `tls` 섹션에 `certResolver` 등을 직접 입력하도록 커스텀 필드 제공.  
- **버전 관리·백업**  
  - 파일 기반이므로 Git 등 버전 관리 시스템에 `trm-*.yml` 디렉터리를 추가해 변경 이력을 추적한다.  
  - 정기적인 디렉터리 스냅샷(예: `rsync` 또는 `restic`)을 권장한다.  
- **모바일·반응형 UI**  
  - UI는 반응형 디자인을 적용해 스마트폰에서도 라우트 추가·수정이 가능하다.  

## 보안 및 인증
- **토큰 기반 인증**: `AUTH_TOKEN`을 환경변수로 설정하고, UI 로그인 및 API 호출 시 동일 토큰을 사용한다. 토큰은 최소 32자 이상의 난수 문자열을 권장한다.  
- **파일 접근 권한**: `CONFIG_DIR` 디렉터리는 `600`(파일) / `700`(디렉터리) 권한을 부여해 Traefik과 TRM만 읽고 쓸 수 있도록 설정한다.  
- **HTTPS 적용**: TRM 자체 UI를 HTTPS로 노출하려면 앞단에 Traefik(또는 Nginx) 리버스 프록시를 두고 TLS 종료를 수행한다.  
- **감시·재로드**: Traefik은 `file` 프로바이더의 `watch: true` 옵션을 사용해 파일 변경 시 자동 재로드한다.  

## 운영 및 유지보수
- **자동 감시·재로드**: `watch: true` 설정이 활성화된 경우, `trm-*.yml` 파일이 변경되면 Traefik이 즉시 라우트를 적용한다.  
- **로그 모니터링**  
  - 컨테이너 로그(`docker logs traefik-route-manager`)에 라우트 생성·삭제 이벤트가 출력된다.  
  - 오류 발생 시 `error` 레벨 로그를 확인하고, 파일 권한·YAML 구문 오류를 점검한다.  
- **업데이트 절차**  
  1. 최신 이미지 확인: `docker pull ghcr.io/jae-jae/traefik-route-manager:main`.  
  2. 기존 컨테이너 정지·삭제: `docker rm -f traefik-route-manager`.  
  3. 새 이미지로 재시작(위 Docker 명령 재사용).  
  - 이미지 교체 시 `CONFIG_DIR` 마운트가 동일하게 유지되는지 확인한다.  

## 비교 분석
| 항목 | Traefik Route Manager (TRM) | Nginx Proxy Manager (NPM) | Traefik Pilot / Hub |
|------|----------------------------|---------------------------|----------------------|
| **데이터베이스** | 없음 (파일 기반) | MySQL 필요 | 클라우드 서비스 (SaaS) |
| **지원 대상** | Traefik File Provider 전용 | Nginx 전용 | Traefik 전체 (클라우드) |
| **배포 무게** | ~15 MB 단일 바이너리 | ~200 MB (PHP+MySQL) | SaaS 구독 |
| **AI 연동** | 내장 Skill·Agents 제공 | 없음 | 제한적 (API) |
| **운영 복잡도** | 낮음 (Docker 마운트만) | 중간 (DB·SSL 설정) | 고도 (클라우드 인증) |

**언제 선택할까?**  
- **경량 UI**가 필요하고 **데이터베이스를 도입하고 싶지 않을 때** → TRM.  
- **Nginx 기반** 환경이거나 **복합적인 인증·SSL 관리**가 필요할 때 → NPM.  
- **대규모 멀티‑클러스터**와 **중앙 관리**가 목표라면 Traefik Hub가 적합.

## 트러블슈팅 FAQ
1. **UI 접속 불가**  
   - 컨테이너 포트(`8892`)가 호스트 방화벽에 차단됐는지 확인.  
   - `docker logs`에 `binding address` 오류가 없는지 점검.  
2. **라우트 파일이 적용되지 않음**  
   - Traefik `file` 프로바이더에 `watch: true`가 설정돼 있는지 확인.  
   - 파일 권한이 `600` 이상인지, Traefik 프로세스가 읽을 수 있는지 점검.  
3. **인증 토큰 오류**  
   - UI 로그인 시 입력한 토큰과 `AUTH_TOKEN` 환경변수가 정확히 일치하는지 확인.  
   - 토큰에 공백이나 특수문자가 포함돼 있으면 URL 인코딩 문제 발생 가능.  
4. **Docker 환경에서 메모리 부족**  
   - 이미지 자체는 가볍지만, 동시에 여러 라우트 파일을 마운트할 경우 호스트 디스크 I/O를 모니터링한다.  

## 결론
- **TRM**은 Traefik File Provider 사용 시 발생하는 **수동 YAML 편집** 문제를 **파일 기반 UI** 하나로 해결한다.  
- **핵심 효과**: 설정 오류 감소, 라우트 추가·수정 시간 단축, AI 에이전트 연동을 통한 자동화 가능.  
- **확장성**: 파일 기반이므로 Git·CI와 연계해 버전 관리·배포 파이프라인에 쉽게 통합 가능.  
- **향후 로드맵**: 커뮤니티 기여를 통한 플러그인·템플릿 추가, 더 풍부한 AI 스킬 지원, 멀티‑엔트리포인트 UI 개선이 예정이다.
