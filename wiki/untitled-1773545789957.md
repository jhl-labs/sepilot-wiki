## 1. 문서 개요
### 목적 및 대상 독자
이 문서는 `services/search.ts` 모듈을 이용해 위키 검색 API와 연동하는 방법을 설명합니다.  
대상은 백엔드 개발자, API 설계자, 운영 담당자 등 검색 기능을 구현·운용해야 하는 모든 팀원입니다.

### 검색 서비스의 역할 및 비즈니스 가치
검색 서비스는 사용자 입력(`q`)을 위키 검색 API에 전달하고, 결과를 가공·캐시하여 빠른 응답을 제공합니다.  
이를 통해 **콘텐츠 탐색 효율성 향상**, **트래픽 비용 절감**, **검색 경험 일관성**을 달성할 수 있습니다.

### 문서 구성 및 활용 방법
- **시스템 아키텍처**: 모듈 위치와 외부 의존 관계  
- **API 정의**: 엔드포인트, 인증, 응답 스키마  
- **파라미터·페이징·캐시**: 사용 규칙과 설계 원칙  
- **오류 처리·테스트·운영**: 실전 적용 시 참고 사항  
각 섹션을 순차적으로 읽으면 서비스 구현 → 테스트 → 배포까지 전 과정을 따라갈 수 있습니다.

## 2. 시스템 아키텍처
### `services/search.ts` 위치와 책임
- **경로**: `services/search.ts`  
- **주요 책임**  
  1. HTTP 요청을 받아 위키 검색 API에 전달  
  2. 쿼리 파라미터 검증 및 정규화  
  3. 결과 캐싱(메모리·Redis) 및 TTL 관리  
  4. 응답 포맷 변환 및 메타데이터 추가  

> 구체적인 함수·클래스 구조는 현재 레포지토리 코드를 검토해야 합니다. **추가 조사가 필요합니다**.

### 위키 검색 API 연동 흐름
1. 클라이언트 → 검색 서비스 엔드포인트 (HTTP)  
2. 서비스 → 파라미터 검증 → 캐시 키 생성  
3. 캐시 히트 → 저장된 결과 반환 (TTL 미만)  
4. 캐시 미스 → 위키 검색 API 호출 → 응답 저장 후 반환  

### 주요 의존 모듈 및 외부 서비스
- **HTTP 클라이언트** (예: `axios` 혹은 `node-fetch`) – 위키 API 호출  
- **캐시 레이어**: 인메모리(LRU) 및 Redis (공통 캐시 인터페이스)  
- **로깅**: `winston`·`pino` 등 구조화 로그  
- **인증/인가**: JWT 혹은 API 키 검증 미들웨어  

> 실제 사용 중인 라이브러리 버전·구성은 코드베이스를 확인해야 합니다. **추가 조사가 필요합니다**.

## 4. 쿼리 파라미터
- **`q`** (검색어) – *필수*  
  - URL 인코딩 필요 (`encodeURIComponent`)  
- **`type`** (검색 대상) – *옵션*  
  - 허용 값: `article`, `user`, `file` 등 (정확한 목록은 API 문서 참고)  
- **`sort`** (정렬 기준) – *옵션*  
  - 기본값: `relevance`  
  - 지원 옵션: `date`, `title` 등  
- **기타 필터**  
  - `category` – 카테고리 제한  
  - `dateRange` – `startDate,endDate` 형식 (ISO 8601)  

> 각 파라미터의 유효성 검사 로직은 `services/search.ts`에 구현되어 있어야 합니다. **추가 조사가 필요합니다**.

## 5. 페이징 전략
- **`page`**: 1 기반 페이지 번호, 기본값 `1`  
- **`size`**: 페이지당 항목 수, 기본값 `20`  
- **최대 페이지 크기**: `100` (초과 시 제한) – 실제 제한값은 코드 확인 필요  
- **메타 데이터**: `totalCount`, `hasNext`를 응답에 포함하여 클라이언트가 다음 페이지 존재 여부를 판단하도록 함  

> 페이징 구현 방식(오프셋 vs. 커서) 여부는 현재 코드에 따라 다를 수 있습니다. **추가 조사가 필요합니다**.

## 6. 캐싱 전략
### 적용 범위
- **읽기 전용 검색 결과** 전체 혹은 일부(예: `type=article`만) 캐시  
- **키 설계**: `search:{q}:{type}:{sort}:{page}:{size}` 형태, 파라미터 순서와 인코딩 규칙을 동일하게 유지  

### TTL 설정 기준
- 일반 검색: `5분` ~ `15분` 사이 (검색어 변동성에 따라 조정)  
- 인기 검색어 또는 최신 업데이트가 빈번한 경우 TTL을 짧게 설정  

### 무효화 정책
- **검색어 업데이트**: 해당 키 전체 삭제 혹은 TTL 만료 대기  
- **API 버전 변경**: 버전 접두사(`v1`)를 키에 포함해 자동 격리  

> 실제 TTL 값·키 정책은 서비스 운영 정책에 따라 달라집니다. **추가 조사가 필요합니다**.

## 7. 오류 처리 및 응답 코드
### 공통 오류 구조
```json
{
  "error": {
    "code": string,
    "message": string,
    "details": any
  }
}
```
### 주요 HTTP 상태 코드 매핑
- **400 Bad Request** – 파라미터 검증 실패 (`INVALID_PARAMETER`)  
- **401 Unauthorized** – 인증 토큰 누락·만료 (`UNAUTHORIZED`)  
- **404 Not Found** – 위키 API에서 결과 없음 (`NO_RESULTS`)  
- **429 Too Many Requests** – 레이트 제한 초과 (`RATE_LIMIT_EXCEEDED`)  
- **500 Internal Server Error** – 서버 내부 오류 (`INTERNAL_ERROR`)  

### 재시도 정책 및 백오프
- **429** 및 **5xx** 응답에 대해 **지수 백오프**(예: 100ms → 200ms → 400ms) 적용  
- 재시도 횟수는 **3회** 이하로 제한 (구현 시 설정값 확인 필요)  

> 구체적인 에러 코드 문자열 및 재시도 로직은 현재 구현을 검토해야 합니다. **추가 조사가 필요합니다**.

## 8. 사용 예시
### cURL 요청 예시
```bash
curl -G "https://api.example.com/api/search" \
  --data-urlencode "q=SePilot" \
  --data "type=article" \
  --data "page=1" \
  --data "size=20" \
  -H "Authorization: Bearer <TOKEN>"
```

### JavaScript/TypeScript 클라이언트 샘플
```typescript
import axios from 'axios';

interface SearchResultItem {
  title: string;
  snippet: string;
  url: string;
  category: string;
  updatedAt: string;
}

interface SearchResponse {
  totalCount: number;
  page: number;
  size: number;
  hasNext: boolean;
  items: SearchResultItem[];
  cache: { hit: boolean; ttl: number };
}

export async function search(query: string, opts?: {
  type?: string;
  sort?: string;
  page?: number;
  size?: number;
}): Promise<SearchResponse> {
  const params = new URLSearchParams({
    q: query,
    ...(opts?.type && { type: opts.type }),
    ...(opts?.sort && { sort: opts.sort }),
    ...(opts?.page && { page: String(opts.page) }),
    ...(opts?.size && { size: String(opts.size) })
  });

  const resp = await axios.get<SearchResponse>('/api/search', {
    params,
    headers: { Authorization: `Bearer ${process.env.API_TOKEN}` }
  });
  return resp.data;
}
```

### 페이지네이션 및 캐시 활용 시나리오
1. 첫 페이지 요청 → 캐시 미스 → API 호출 → 결과 저장(TTL 10분)  
2. 동일 파라미터 재요청 → 캐시 히트 → `cache.hit:true` 반환, 응답 시간 단축  
3. 다음 페이지(`page=2`) 요청 → 새로운 캐시 키 생성 → 동일 흐름 반복

## 11. FAQ
**Q1. `q` 파라미터에 특수 문자를 넣으면 오류가 발생합니다.**  
A. 반드시 `encodeURIComponent` 로 인코딩하고, 서버 측에서도 디코딩 검증을 수행합니다.

**Q2. 페이지 크기를 크게(`size=200`) 요청하면 400 오류가 반환됩니다.**  
A. 현재 서비스는 최대 `size=100`으로 제한하고 있습니다. 제한값은 구현을 확인해 주세요. **추가 조사가 필요합니다**.

**Q3. 캐시가 자주 무효화돼서 성능이 저하됩니다.**  
A. TTL 설정이 너무 짧을 수 있습니다. 운영 정책에 맞게 TTL을 조정하고, 검색어 업데이트 빈도와 연동해 보세요.

**Q4. 레이트 제한(429) 발생 시 재시도 로직이 동작하지 않습니다.**  
A. 클라이언트 측에서 지수 백오프를 구현했는지 확인하고, `Retry-After` 헤더를 활용하도록 코드를 수정합니다.
