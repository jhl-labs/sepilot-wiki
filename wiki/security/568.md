## 1. 서론
이 문서는 npm 레지스트리를 통한 공급망 공격 사례를 이해하고, 조직·개인 개발자가 동일한 위협에 대비할 수 있도록 실무 가이드를 제공한다.  
대상 독자는 **보안 엔지니어, DevOps 팀, 개발자, CI/CD 운영자**이며, npm 기반 프로젝트를 운영하는 모든 환경에 적용 가능하다.  

최근 **npm 유지보수자 계정 탈취**와 **자격 증명·API 키 탈취**를 목표로 한 두 악성 패키지(`sbx-mask`, `touch-adv`)가 보고되었다(DevOps.com, Sonatype)【https://euno.news/posts/ko/two-malicious-npm-packages-aim-to-steal-credential-743e9d】. 본 문서는 해당 사건을 기반으로 최신 npm 공급망 공격 동향을 정리하고, 탐지·완화·사고 대응 방안을 제시한다.

## 2. 위협 개요
### 악성 npm 패키지 정의 및 특징
npm 레지스트리에 게시된 패키지는 일반적으로 오픈소스 라이브러리·툴을 제공한다. 악성 패키지는 **정상적인 기능을 가장하거나, 설치 시 자동으로 악성 코드를 실행**하여 공격자의 목적을 달성한다. 주요 특징은 다음과 같다.  

- **계정 탈취 기반 배포**: 공격자는 유지보수자(패키지 소유자) 계정을 탈취해 패키지를 공식 레지스트리에 게시한다.  
- **시크릿 수집 로직 포함**: 설치 과정에서 로컬 환경에 저장된 자격 증명, API 키, `.npmrc`, `git-config` 등 다양한 시크릿을 탐색·수집한다.  
- **무차별 배포**: 의존성 트리 상위에 위치한 패키지로 위장해 다수 프로젝트에 전파될 가능성이 있다.  

### sbx‑mask와 touch‑adv 패키지 개요
- **sbx-mask**: Sonatype 보안 연구팀이 식별한 악성 npm 패키지 중 하나. 구체적인 동작 방식은 공개되지 않았으며, 자격 증명·시크릿 탈취를 목표로 한다.  
- **touch‑adv**: 동일 연구팀이 보고한 두 번째 악성 패키지. 역시 설치 시 로컬에 저장된 비밀 정보를 탈취하도록 설계되었다.  

> **※ 위 두 패키지의 상세 코드·동작 흐름은 현재 공개된 자료에 포함되지 않아 추가 조사가 필요합니다.**  

### 공격자 목표
- **자격 증명·API 키 탈취**: npm 레지스트리·CI/CD 파이프라인에 사용되는 토큰·키를 획득해 클라우드·서비스에 무단 접근.  
- **추가 악성 행위**: 탈취된 시크릿을 이용해 추가 악성 패키지 배포·인프라 장악 등 2차 공격 수행 가능성.  

## 3. 공격 시나리오 상세 분석
### npm 유지보수자 계정 탈취 경로
공격자는 **비밀번호 유출, 피싱, 2FA 우회** 등을 통해 npm 계정을 장악한다. 구체적인 탈취 방법은 보고서에 명시되지 않아 **추가 조사가 필요**합니다.

### 악성 패키지 배포·공개 과정
탈취된 계정으로 npm 레지스트리에 `sbx-mask`, `touch-adv`를 **정상 패키지처럼** 게시한다. 이후 개발자가 해당 패키지를 의존성에 추가하면 자동으로 설치된다.

### 패키지 설치 시 실행되는 악성 코드 흐름
- `npm install` 시 **postinstall 스크립트** 혹은 **preinstall 스크립트**가 실행되어 로컬 파일·환경 변수를 탐색한다.  
- 탐색 대상은 **환경 변수(`process.env`), `.npmrc`, `git-config`, 홈 디렉터리 파일** 등이다.  
- 수집된 시크릿은 **외부 서버**(예: 공격자 제어 C2)로 전송된다.  

> **※ 실제 스크립트 내용·전송 방식은 공개되지 않아 추가 조사가 필요합니다.**  

### 대상 시스템에서 시크릿 수집 방법
- **환경 변수**: `NPM_TOKEN`, `AWS_ACCESS_KEY_ID`, `GITHUB_TOKEN` 등.  
- **설정 파일**: `~/.npmrc`에 저장된 레지스트리 토큰, `~/.gitconfig`에 포함된 인증 정보.  
- **프로젝트 파일**: `.env`, `config.js` 등에 하드코딩된 비밀.  

## 4. 영향 평가
### 잠재적 피해 범위
- **개인 개발자**: 개인 계정·클라우드 리소스 탈취.  
- **기업**: CI/CD 파이프라인에 저장된 토큰·키가 유출돼 **빌드·배포 자동화**가 공격자에게 노출될 위험.  
- **공급망 전체**: 악성 패키지가 다수 프로젝트에 전파되면 **연쇄적인 시크릿 유출**이 발생할 수 있다.  

### 탈취된 시크릿이 초래할 수 있는 보안 위험
- **API 남용**: 클라우드 서비스·외부 API 무단 호출.  
- **데이터 유출**: 데이터베이스·스토리지 접근을 통한 민감 정보 탈취.  
- **권한 상승**: 탈취된 토큰을 이용해 추가 인프라·서비스에 대한 관리자 권한 획득.  

### 실제 사례 및 피해 보고서 요약
현재 공개된 자료에서는 **구체적인 피해 사례**(예: 특정 기업의 데이터 유출)와 **피해 규모**에 대한 언급이 없으며, 이는 **추가 조사가 필요**합니다.

## 5. 탐지 방법
### npm 레지스트리 모니터링 포인트
- **패키지 메타데이터**: 신규 패키지·버전 급증, 유지보수자 계정 생성·변경 이력.  
- **유지보수자 활동**: 비정상적인 로그인·IP 변화, 2FA 비활성화 여부.  

### 로컬 개발 환경에서의 이상 행위 탐지
- **파일 시스템**: `node_modules` 내 **postinstall** 스크립트 파일 생성·수정 감시.  
- **네트워크 트래픽**: 의심스러운 외부 C2 서버로의 HTTP/HTTPS 요청 탐지.  
- **프로세스 실행**: `npm install` 후 자동 실행되는 스크립트 로그 분석.  

### 기존 시크릿 스캐너·SAST 도구와의 연계 방안
- **Snyk**, **npm audit**, **GitHub Dependabot** 등 의존성 스캐너와 **Sonatype Nexus IQ**를 연동해 악성 패키지 식별.  
- **Secret Scanning**(GitHub, GitLab)으로 레포지토리 내 시크릿 노출 여부 사전 차단.  

## 6. 완화 및 방어 전략
### npm 계정 보안 강화
- **2FA(다중 인증) 활성화** 및 **보안 키** 사용.  
- **비밀번호 정책**: 주기적 변경·복잡도 유지.  
- **최소 권한 원칙** 적용: 조직 계정·CI 토큰에 제한된 권한 부여.  

### 패키지 검증 절차
- **SHA‑256 해시** 확인 및 **npm 패키지 서명**(npm v8 이상) 검증.  
- **신뢰할 수 있는 레지스트리**(예: npmjs.com 공식 레지스트리) 사용 및 사설 레지스트리 도입 시 서명 정책 적용.  

### CI/CD 파이프라인에서의 공급망 보안 정책
- **npm audit** 실행 및 **취약점 자동 업데이트**(Dependabot, Renovate).  
- **Snyk**, **Sonatype** 등 서드파티 보안 도구와 연동해 의존성 스캔 자동화.  

### 시크릿 관리 베스트 프랙티스
- **환경 변수 최소화** 및 **시크릿 매니저**(AWS Secrets Manager, HashiCorp Vault) 사용.  
- **.npmrc**, **.env** 파일을 **gitignore**에 포함하고, 레포지토리 내 시크릿 커밋 방지.  

## 7. 사고 대응 가이드
### 악성 패키지 발견 시 즉각적인 조치 단계
1. **패키지 설치 중단**: `npm uninstall` 및 CI 파이프라인 중단.  
2. **시크릿 회수**: 노출된 토큰·키를 즉시 **폐기·재발급**.  
3. **레포지토리 정리**: 악성 패키지와 관련된 커밋을 **리베이스**하거나 **필터링**.  

### 침해된 시크릿 회수 및 재발급 절차
- 해당 서비스(예: GitHub, AWS) 콘솔에서 **토큰 폐기** 후 새 토큰 발급.  
- **CI/CD 변수**와 **환경 변수**를 모두 교체하고, 이전 토큰 사용 로그를 검토한다.  

### npm 레지스트리 신고·패키지 제거 프로세스
- npm 공식 **abuse report** 페이지(https://www.npmjs.com/report-abuse) 에 신고.  
- 유지보수자 계정 복구·비활성화 요청 후, 패키지 **삭제** 절차 진행.  

### 포스트모템 분석 및 재발 방지 계획 수립
- **루트 원인 분석(RCA)** 수행: 계정 탈취 경로·패키지 배포 과정 파악.  
- **보안 정책 업데이트**: 2FA 강제, 패키지 서명 의무화 등.  

## 8. 공급망 보안 향상을 위한 권고사항
### 조직 차원의 npm 공급망 위험 관리 프레임워크
- **위험 식별**: 의존성 트리 분석·위험 등급 부여.  
- **위험 평가**: 정기적인 보안 스캔·감사 수행.  
- **위험 대응**: 고위험 패키지 차단·대체 라이브러리 검토.  

### 정기적인 의존성 검토 및 업데이트 정책
- **월간/주간** 의존성 업데이트 스케줄 수립.  
- **자동화 도구**(Dependabot, Renovate) 활용해 최신 버전 유지.  

### 서드파티 패키지 신뢰도 평가 모델 도입
- **패키지 메타데이터**(다운로드 수, 유지보수자 활동)와 **보안 스코어**(Snyk, Sonatype) 기반 평가.  

### 커뮤니티와 협업하여 악성 패키지 조기 탐지·공유
- **OSS 보안 커뮤니티**(GitHub Security Lab, npm Security)와 정보 공유.  
- **공개 보고서**(Sonatype, Snyk) 활용해 최신 위협 인텔리전스 반영.  

## 9. 결론
- **주요 교훈**: npm 유지보수자 계정 탈취를 통한 공급망 공격은 **시크릿 탈취**라는 심각한 위험을 동반한다.  
- **예방이 최선**: 계정 보안 강화·패키지 검증·CI/CD 보안 자동화가 핵심 방어선이다.  
- **지속적인 모니터링**: npm 레지스트리·의존성 변화를 실시간으로 감시하고, 커뮤니티와 협업해 새로운 악성 패키지를 조기에 차단해야 한다.
