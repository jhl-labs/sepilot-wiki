# 1. 개요
Telegram 애플리케이션에 **OpenID Connect (OIDC)** 와 **Proof Key for Code Exchange (PKCE)** 를 적용해 **전용 Access Layer** 를 구현하는 방법을 안내합니다.  

대상 독자  

* 백엔드·프론트엔드 개발자  
* 보안·인프라 엔지니어  
* 인증·인가 설계 담당자  

기존 세션‑쿠키·패스워드 기반 인증을 표준화된 프로토콜 기반 모델로 전환하고자 하는 팀에 적합합니다.

## 1.1 필요성
Telegram 로그인은 **Telegram Login Widget**(또는 Bot API) 를 통해 사용자를 인증합니다. 이를 OIDC + PKCE 로 감싸면  

* **스코프·수명** 을 세밀하게 제어  
* **Trusted Client Confirmation** 으로 클라이언트 위조 방지  
* **토큰 기반** 접근 제어로 확장성 확보  

가 가능합니다.

## 1.2 주요 용어
| 용어 | 정의 |
|------|------|
| **OIDC** | OAuth 2.0 위에 구축된 인증 표준. ID Token(JWT) 으로 사용자 신원을 증명 |
| **PKCE** | 공개 클라이언트에서 Authorization Code 탈취를 방지하는 `code_verifier` / `code_challenge` 메커니즘 |
| **Access Layer** | 토큰 발급·검증·폐기·키 회전을 담당하는 전용 서비스 |
| **Scope** | 클라이언트가 요청할 수 있는 권한 집합(예: `phone_number`, `messages:read`) |
| **Trusted Client Confirmation** | 앱 서명·패키지명 등 메타데이터를 검증해 클라이언트 위조를 차단 |

# 2. Telegram 로그인 흐름 (실제 구현)

Telegram은 전통적인 OAuth 2.0 엔드포인트를 제공하지 않으며, **Login Widget** 을 이용해 사용자를 인증하고, **Bot API** 로 사용자 정보를 전달받습니다. 아래는 OIDC + PKCE 로 감싼 전체 흐름입니다.

```
1. 클라이언트 → Telegram Login Widget (https://telegram.org/js/telegram-widget.js) 요청
   - 파라미터: bot_id, redirect_uri, state, nonce, pkce_challenge
2. 사용자는 Telegram 앱(또는 웹)에서 로그인·동의
3. Telegram → redirect_uri (앱) 으로
   - auth_date, hash, user (id, first_name, …), state, nonce, pkce_challenge
4. 앱 서버는 받은 파라미터와 Bot API `getChatMember` 로 사용자 검증
5. 검증 성공 → 내부 OIDC Authorization Server에 Authorization Code 발행
6. 클라이언트는 PKCE `code_verifier` 로 토큰 교환
7. Access Token / ID Token 반환 → Access Layer 로 전달
```

> **핵심 포인트**  
> * `state` 와 `nonce` 로 CSRF·재생 공격 방지  
> * `pkce_challenge`(=`code_challenge`) 를 Widget 파라미터에 포함해 PKCE 를 완전 적용  

## 2.1 Telegram Login Widget 설정 예시
```text
https://telegram.org/js/telegram-widget.js?5
  &bot_id=123456789
  &redirect_uri=https%3A%2F%2Fyourapp.com%2Ftelegram%2Fcallback
  &state=RANDOM_STATE
  &nonce=RANDOM_NONCE
  &pkce_challenge=BASE64URL(SHA256(code_verifier))
  &request_access=write
```

## 2.2 서버‑사이드 검증 흐름 (Node.js 예시)

```javascript
// 1) Callback 파라미터 파싱
const url = new URL(request.url);
const params = Object.fromEntries(url.searchParams.entries());

// 2) state 검증 (세션에 저장된 값과 비교)
if (params.state !== request.session.oauthState) {
  throw new Error('Invalid state');
}

// 3) nonce 검증 (ID Token 검증 시 사용)
const receivedNonce = params.nonce;

// 4) Telegram 해시 검증 (Telegram 문서에 명시된 HMAC‑SHA256)
const crypto = require('crypto');
const secret = crypto.createHash('sha256')
                     .update(process.env.TELEGRAM_BOT_TOKEN)
                     .digest();
const dataCheckString = Object.keys(params)
  .filter(k => k !== 'hash')
  .sort()
  .map(k => `${k}=${params[k]}`)
  .join('\n');
const hash = crypto.createHmac('sha256', secret)
                   .update(dataCheckString)
                   .digest('hex');
if (hash !== params.hash) {
  throw new Error('Telegram data integrity check failed');
}

// 5) Bot API 로 사용자 존재 여부 확인 (선택)
await fetch(`https://api.telegram.org/bot${process.env.TELEGRAM_BOT_TOKEN}/getChatMember`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ chat_id: params.id, user_id: params.id })
});

// 6) 내부 OIDC Authorization Server에 Authorization Code 발행
//    (PKCE code_challenge 를 그대로 전달)
```

# 3. OIDC + PKCE 핵심 설계

## 3.1 Authorization Request (PKCE 포함)

```
GET https://auth.example.com/authorize?
    response_type=code&
    client_id=YOUR_CLIENT_ID&
    redirect_uri=https%3A%2F%2Fyourapp.com%2Fcallback&
    scope=openid%20profile%20phone_number&
    state=RANDOM_STATE&
    nonce=RANDOM_NONCE&
    code_challenge=BASE64URL(SHA256(code_verifier))&
    code_challenge_method=S256
```

* `code_verifier` : 43~128자 ASCII 랜덤 문자열  
* `code_challenge` : `BASE64URL( SHA256(code_verifier) )`  

## 3.2 Token Request

```
POST https://auth.example.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&
code=AUTH_CODE&
redirect_uri=https%3A%2F%2Fyourapp.com%2Fcallback&
client_id=YOUR_CLIENT_ID&
code_verifier=ORIGINAL_CODE_VERIFIER
```

## 3.3 ID Token 검증 절차
1. `iss`, `aud`, `exp`, `iat` 검증  
2. `nonce` 가 최초 요청 시 저장한 값과 일치하는지 확인  
3. `state` 가 세션에 저장된 값과 일치하는지 확인 (CSRF 방어)  
4. JWKS 엔드포인트에서 공개키를 받아 서명 검증 (`RS256` 권고)

# 4. Access Layer 설계 (중복 제거 및 통합)

| 구성 요소 | 역할 | 주요 설정 |
|-----------|------|-----------|
| **Authorization Server** | OIDC + PKCE 흐름 관리 | `code_challenge_method=S256`, `redirect_uri` 엄격 관리 |
| **Token Service** | Access/Refresh Token 발급·검증·폐기 | Access Token TTL 5‑15분, Refresh Token TTL 30일 |
| **Scope Registry** | 비즈니스 스코프 정의·검증 | 최소 권한 원칙 적용, 스코프‑클레임 매핑 |
| **Trusted Client Validator** | 앱 서명·패키지명 검증 | Bot API 로 전달된 메타데이터와 사전 등록 값 비교 |
| **Key Management** | JWKS 제공·키 회전 | 키 회전 주기 최소 90일, 자동 배포 파이프라인 연동 |
| **Audit Logger** | 토큰 발급·폐기·리프레시 로그 | ELK/Loki 로 중앙 수집, 이상 징후 탐지 규칙 적용 |

> **주의**: 토큰 수명·키 회전 정책은 조직 보안 기준에 맞게 조정하십시오.

# 5. 보안 강화 상세

## 5.1 `state`·`nonce` 생성·검증 예시 (Node.js)

```javascript
// 생성 (로그인 시작 시)
const crypto = require('crypto');
const state = crypto.randomBytes(16).toString('hex');
const nonce = crypto.randomBytes(16).toString('hex');
request.session.oauthState = state;   // 세션에 저장
request.session.oauthNonce = nonce;

// 검증 (콜백 처리 시)
if (params.state !== request.session.oauthState) throw new Error('Invalid state');
if (idTokenPayload.nonce !== request.session.oauthNonce) throw new Error('Invalid nonce');
```

* `state` : CSRF 방지, 세션에 저장 후 비교  
* `nonce` : 재생 공격 방지, ID Token 클레임에 포함  

## 5.2 토큰 폐기·리프레시 정책

| 상황 | 처리 |
|------|------|
| **Access Token 만료** | 클라이언트는 Refresh Token 으로 새 Access Token 요청 |
| **Refresh Token 사용** | 새 Refresh Token 발급 후 기존 Refresh Token 즉시 폐기 (rotate) |
| **사용자 로그아웃 / 계정 비활성화** | 모든 Access·Refresh Token 즉시 폐기 (`revocation endpoint` 호출) |
| **키 회전** | 기존 토큰 검증 실패 시 자동 폐기, 재인증 유도 |

### 토큰 회전 구현 흐름 (pseudo)

```text
1. 클라이언트 → /token (grant_type=refresh_token, refresh_token=OLD)
2. 서버:
   a) OLD 토큰 검증 → 유효하면 새로운 Access + Refresh 발행
   b) OLD 토큰을 DB/Cache 에서 삭제 (폐기)
3. 클라이언트는 새로운 Refresh Token 만을 저장
```

## 5.3 키 관리 및 회전

1. JWKS 엔드포인트(`/.well-known/jwks.json`) 에 현재 활성 키와 이전 키 2개 제공  
2. 키 회전 시  
   * 새 키를 생성 → JWKS에 추가 → 기존 키는 `inactive` 플래그 설정 후 30일간 유지  
   * 토큰 검증 로직은 `kid` 로 현재/이전 키를 모두 시도  

# 6. 테스트·검증 체크리스트

| 테스트 항목 | 기대 결과 |
|-------------|-----------|
| **Authorization → Token 흐름** | 정상적인 Authorization Code → Access Token 발급 |
| **PKCE 검증** | `code_verifier` 불일치 시 토큰 발급 실패 |
| **state·nonce 검증** | 변조된 `state`/`nonce` 요청 시 400 오류 |
| **Trusted Client Confirmation** | 등록되지 않은 앱 서명·패키지명으로는 토큰 발급 안 됨 |
| **Scope 매핑** | 요청 스코프와 토큰 `scope` 클레임이 일치 |
| **Refresh Token 회전** | 재발급 시 기존 Refresh Token 폐기 확인 |
| **키 회전** | 이전 키로 서명된 토큰은 30일 이내에만 검증 가능 |

자동화 도구: `Postman` 컬렉션, `pytest`(Python) + `requests`, `JUnit`(Java) 등 CI 파이프라인에 포함.

# 8. 결론 및 향후 로드맵

* **Access Layer** 를 도입하면 Telegram 로그인뿐 아니라 다양한 외부 인증 제공자를 OIDC + PKCE 로 일관되게 통합할 수 있습니다.  
* `state`, `nonce`, `PKCE` 로 CSRF·재생·코드 탈취 위험을 최소화하고, **Trusted Client Confirmation** 으로 클라이언트 위조를 방지합니다.  
* 향후 **OAuth 2.1**, **DPoP**, **OAuth Device Flow** 등 새로운 표준이 등장해도 현재 설계는 **확장 가능** 하도록 모듈화되었습니다.
