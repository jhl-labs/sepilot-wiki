## 1. 서론
멀티‑테넌트 SaaS 환경에서는 **테넌트 간 데이터 격리**가 가장 중요한 보안 요구사항 중 하나입니다.  
전통적으로는 애플리케이션 레이어에서 `tenant_id` 필터를 모든 쿼리에 삽입하고, 미들웨어와 단위 테스트로 검증해 왔습니다. 그러나 다음과 같은 상황에서 이 접근법은 **취약점**을 남깁니다.

- 개발 실수로 `WHERE tenant_id = …` 조건이 누락  
- 악의적인 스크립트가 실행돼 필터를 재정의  
- 운영팀이 직접 DB 관리 도구나 ad‑hoc 쿼리로 접근할 경우 필터가 적용되지 않음  

이러한 위험을 근본적으로 차단하려면 **보안 경계를 애플리케이션 코드가 아닌 데이터베이스 자체**에 두어야 합니다. 본 문서는 PostgreSQL의 Row‑Level Security(RLS)를 활용해 멀티‑테넌트 격리를 구현하고, 이를 SQLAlchemy와 Alembic 워크플로에 자동화하는 방법을 단계별로 제시합니다. (출처: euno.news [링크](https://euno.news/posts/ko/stop-leaking-data-in-multi-tenant-apps-ebfe5f))

## 2. 애플리케이션 레이어 보안의 취약점
| 취약 시나리오 | 발생 원인 | 결과 |
|---|---|---|
| `tenant_id` 필터 누락 | 개발자 실수, 코드 리팩터링 | 전체 테넌트 데이터 노출 |
| 필터 재정의 | 악의적 스크립트, 동적 SQL | 권한 없는 데이터 접근 |
| 직접 DB 접속 | 관리자 도구, ad‑hoc 쿼리 | RLS 정책 미적용 → 데이터 유출 |

위와 같은 상황은 **애플리케이션 레이어만으로는 방어가 불가능**함을 보여줍니다.

## 3. Row‑Level Security(RLS) 기본 개념
- **정의**: PostgreSQL이 제공하는 행 수준 접근 제어 기능으로, 테이블에 적용된 정책에 따라 사용자가 볼 수 있는 행을 제한합니다.  
- **작동 원리**: 세션 변수(`SET LOCAL`)에 현재 테넌트 ID를 저장하고, `USING` 절에 정의된 조건과 매칭되는 행만 반환합니다.  
- **정책 유형**  
  - **PERMISSIVE**: 기존 정책에 추가로 허용 행을 정의  
  - **RESTRICTIVE**: 접근 가능한 정확한 행 집합을 정의 (보안 모델에 따라 선택)  

## 4. 데이터베이스 설계 변경
1. **`tenant_id` 컬럼 추가**  
   - 모든 멀티‑테넌트 테이블에 `tenant_id UUID NOT NULL` 컬럼을 두고, 테넌트 식별에 사용합니다.  
2. **인덱스 권고**  
   - `tenant_id`에 B‑tree 인덱스를 생성해 RLS 프레디케이트와 조인 성능을 최적화합니다.  
3. **RLS 활성화**  
   ```sql
   ALTER TABLE orders ENABLE ROW LEVEL SECURITY;
   ```
4. **정책 정의 예시**  
   ```sql
   CREATE POLICY tenant_isolation ON orders
   USING (tenant_id = current_setting('app.current_tenant')::uuid);
   ```

## 5. SQLAlchemy와 Alembic을 이용한 자동화
### 5.1 Alembic 마이그레이션 흐름
```python
from alembic import op
import sqlalchemy as sa

def upgrade():
    # tenant_id 컬럼이 없을 경우 추가
    op.add_column('orders', sa.Column('tenant_id', sa.UUID(), nullable=False))
    op.create_index('ix_orders_tenant_id', 'orders', ['tenant_id'])

    # RLS 활성화 및 정책 생성
    op.execute("ALTER TABLE orders ENABLE ROW LEVEL SECURITY;")
    op.execute("""
        CREATE POLICY tenant_isolation ON orders
        USING (tenant_id = current_setting('app.current_tenant')::uuid);
    """)

def downgrade():
    # 정책 삭제 및 RLS 비활성화
    op.execute("DROP POLICY IF EXISTS tenant_isolation ON orders;")
    op.execute("ALTER TABLE orders DISABLE ROW LEVEL SECURITY;")
    op.drop_index('ix_orders_tenant_id', table_name='orders')
    op.drop_column('orders', 'tenant_id')
```
> 위 스크립트는 **멱등(idempotent)**하게 설계되어 CI/CD 파이프라인에 그대로 포함할 수 있습니다. (출처: euno.news)

### 5.2 SQLAlchemy 세션에 테넌트 컨텍스트 주입
```python
import contextvars
from sqlalchemy import event
from sqlalchemy.orm import Session

# 현재 테넌트 UUID를 보관하는 ContextVar
current_tenant = contextvars.ContextVar("current_tenant")

def set_tenant(tenant_id: str):
    current_tenant.set(tenant_id)

@event.listens_for(Session, "before_flush")
def apply_tenant_setting(session, flush_context, instances):
    tenant = current_tenant.get(None)
    if tenant:
        session.execute(
            f"SET LOCAL app.current_tenant = '{tenant}'"
        )
```
각 HTTP 요청 시작 시 `set_tenant()`를 호출하면 이후 실행되는 모든 ORM 쿼리가 자동으로 RLS 정책을 상속받습니다. (출처: euno.news)

## 6. 애플리케이션 미들웨어와 테넌트 설정
- **FastAPI 예시**  
  ```python
  from fastapi import FastAPI, Request

  app = FastAPI()

  @app.middleware("http")
  async def tenant_middleware(request: Request, call_next):
      tenant_id = request.headers.get("X-Tenant-ID")
      if tenant_id:
          set_tenant(tenant_id)
      response = await call_next(request)
      return response
  ```
- **Express (Node.js) 예시** (개념 동일, `setTenant` 함수 호출)  
- **슈퍼유저 정책**  
  - 슈퍼유저는 `app.current_tenant`를 설정하지 않아도 모든 행에 접근 가능하도록 별도 `CREATE POLICY super_user ON … USING (true);` 를 정의합니다.  

## 7. 성능 및 인덱싱 고려사항
- **인덱스**: `tenant_id` 컬럼에 인덱스를 생성하면 RLS 프레디케이트가 인덱스 스캔을 활용해 전체 테이블 스캔을 방지합니다.  
- **쿼리 플랜 변화**: RLS 적용 시 `EXPLAIN` 결과에 `Filter: (tenant_id = $1)` 형태의 필터가 추가됩니다.  
- **대규모 테이블 테스트**: 10M 행 이상 테이블에 대해 `SELECT COUNT(*)` 테스트를 수행해 인덱스 유무에 따른 응답 차이를 측정합니다. (구체적인 수치는 실제 환경에서 측정 필요)

## 8. 테스트·CI/CD 파이프라인 통합
1. **단위 테스트**  
   - `set_tenant()` 후 ORM 쿼리를 실행해 반환되는 행이 해당 테넌트만 포함되는지 검증합니다.  
2. **통합 테스트**  
   - CI 환경에서 Alembic 마이그레이션을 실행하고, 정책이 정상적으로 적용됐는지 `SELECT * FROM pg_policies WHERE tablename='orders';` 로 확인합니다.  
3. **멱등성 보장**  
   - 마이그레이션 스크립트는 이미 정책이 존재하면 `CREATE POLICY IF NOT EXISTS` 형태로 작성해 재실행 시 오류가 발생하지 않게 합니다.  
4. **정책 검증 자동화**  
   - CI 단계에 `psql -c "\d+ orders"` 명령을 추가해 `rowsecurity` 플래그와 정책 존재 여부를 체크합니다.

## 9. 운영 시 보안 관리 체크리스트
- **슈퍼유저 권한 제한**: RLS를 우회할 수 있는 `SUPERUSER` 역할을 최소화하고, 필요 시 별도 감사 로그를 활성화합니다.  
- **정책 변경 감사**: `ALTER POLICY` 명령은 `pg_audit` 등 로그 시스템에 기록하도록 설정합니다.  
- **정기 보안 점검**:  
  - 정책 누락 여부 (`SELECT * FROM pg_policies WHERE tablename='…';`)  
  - 인덱스 상태 (`REINDEX` 필요 시 수행)  
  - 침투 테스트를 통해 직접 DB 접속 시 RLS가 정상 작동하는지 검증

- 단계별 도입 로드맵
