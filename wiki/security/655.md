## 1. 문서 개요
**목적** – Claude.ai와 같은 LLM 커넥터가 요구하는 *OAuth 2.1 Protected Resource Metadata* 를 MCP 서버에 제공하지 못하는 상황을 해결하기 위해, `mcp‑gate` 라는 무상태 사이드카 프록시의 설치·운용 방법을 안내한다.  
**대상 독자** – MCP 서버 운영자, DevOps 엔지니어, 보안 담당자 등 인증 인프라와 MCP 프로토콜을 함께 다루는 실무자.

### 핵심 용어 정의
- **OAuth 2.1** – 최신 OAuth 사양, RFC 6749·RFC 6750·RFC 8414·RFC 9449 등을 포함한다.  
- **MCP** – Model‑Control‑Protocol, Claude.ai 등 LLM 서비스와 커스텀 백엔드 간 통신에 사용되는 프로토콜.  
- **RFC 9728** – *OAuth 2.1 Protected Resource Metadata* 정의, `/.well-known/oauth-protected-resource` 엔드포인트 제공을 규정한다.  
- **JWT** – JSON Web Token, Bearer 토큰 형태로 인증에 사용된다.  
- **JWKS** – JSON Web Key Set, 공개키를 배포해 JWT 서명을 검증한다.  

> 정보는 **EUNO.NEWS** 기사(Dev.to)에서 발췌했습니다 [[출처](https://euno.news/posts/ko/i-couldnt-find-an-oauth-21-proxy-for-mcp-servers-s-439368)].

## 2. 배경 및 동기
Claude.ai 커스텀 커넥터는 다음 흐름을 기대한다.

1. `/.well-known/oauth-protected-resource` 로 메타데이터를 조회한다.  
2. 메타데이터에 명시된 인증 서버와 OAuth 2.1 토큰을 협상한다.  
3. 이후 요청은 `Authorization: Bearer <JWT>` 헤더를 포함하고, 서버는 JWKS 로 서명을 검증한다.

대다수 MCP 서버(Go, Python, TypeScript 등)는 **인증 인프라**가 아니라 **MCP 프로토콜** 자체에 집중하기 때문에 위 과정을 구현하지 않는다 [[출처](https://euno.news/posts/ko/i-couldnt-find-an-oauth-21-proxy-for-mcp-servers-s-439368)].  
따라서 인증 로직을 각 MCP 서버에 직접 삽입하는 것은 설계상의 오류이며, **사이드카 프록시**가 필요하게 된다.

## 3. OAuth 2.1 Protected Resource Metadata (RFC 9728) 개요
- **엔드포인트**: `/.well-known/oauth-protected-resource`  
- **역할**: 클라이언트(Claude.ai 등)에게 인증 서버 위치와 토큰 엔드포인트 등을 알려준다.  
- **필수 항목** (RFC 9728 기준)  
  - `authorization_server` – 인증 서버 베이스 URL  
  - `token_endpoint` – 토큰 교환 엔드포인트  
  - `jwks_uri` – 공개키 배포 URL  
  - 기타 `scopes_supported`, `grant_types_supported` 등  

클라이언트는 이 메타데이터를 기반으로 OAuth 2.1 흐름을 수행한다 [[RFC 9728](https://datatracker.ietf.org/doc/html/rfc9728)].

## 4. 문제 정의
| 문제 | 현상 | 근본 원인 |
|------|------|-----------|
| 메타데이터 엔드포인트 부재 | Claude.ai가 조용히 실패, 오류 로그 없음 | MCP 서버가 RFC 9728 구현을 제공하지 않음 |
| JWT 검증 부재 | Bearer 토큰을 수신해도 검증 로직이 없어 401/403 처리 불가 | 인증 인프라와 MCP 프로토콜이 혼합돼 있지 않음 |
| 인증 로직 중복 | 각 MCP 서버마다 동일 검증 코드 삽입 필요 | 설계상 인증 책임이 분산돼 유지·보수 비용 증가 |

**해결책** – 인증을 담당하는 **사이드카 프록시**(`mcp‑gate`)가 위 세 가지 기능을 일관되게 제공한다 [[출처](https://euno.news/posts/ko/i-couldnt-find-an-oauth-21-proxy-for-mcp-servers-s-439368)].

## 5. mcp‑gate 아키텍처
### 위치 및 역할
```
Claude.ai  →  mcp-gate (Sidecar)  →  실제 MCP 서버
```
- **무상태** Go 바이너리, 어떤 백엔드 언어와도 독립적으로 동작.  
- **역할**  
  1. `/.well-known/oauth-protected-resource` 제공 (RFC 9728 준수)  
  2. 모든 인바운드 요청에 대해 Bearer JWT 검증  
  3. 검증 성공 시 업스트림 MCP 서버로 역방향 프록시, 실패 시 401 반환  

### 주요 컴포넌트
- **메타데이터 제공 모듈** – 환경 변수(`RESOURCE_URI`, `AUTHORIZATION_SERVER`, `JWKS_URI` 등) 기반 정적 JSON 응답 생성.  
- **JWT 검증 모듈** – JWKS를 주기적으로 조회·캐시, 서명(RS256), `iss`, `aud`, `exp`, `nbf`, `scope` 등 클레임 검증.  
- **역방향 프록시 모듈** – Go `httputil.ReverseProxy` 활용, 토큰 유효 시 요청 전달, 무효 시 401 응답.  

> 상세 구현은 `mcp-gate` GitHub 레포지토리(예: `cpremus/mcp-gate`)에서 확인할 수 있습니다 [[출처](https://euno.news/posts/ko/i-couldnt-find-an-oauth-21-proxy-for-mcp-servers-s-439368)].

## 6. 핵심 기능 상세
### 메타데이터 엔드포인트 제공
- **응답 예시** (RFC 9728‑준수)  
  ```json
  {
    "authorization_server": "https://auth.example.com/application/o/mcp/",
    "token_endpoint": "https://auth.example.com/application/o/mcp/token/",
    "jwks_uri": "https://auth.example.com/application/o/mcp/jwks/",
    "scopes_supported": ["mcp.read", "mcp.write"]
  }
  ```
- `RESOURCE_URI` 환경 변수에 지정된 베이스 URL을 사용해 `authorization_server` 등을 구성한다.

### Bearer JWT 검증
- **JWKS 조회·캐시** – `JWKS_URI` 로부터 공개키를 가져와 메모리 캐시, TTL은 내부 기본값(예: 5 분)이며 필요 시 환경 변수로 조정 가능.  
- **검증 대상 클레임**  
  - `iss` → `EXPECTED_ISSUER`와 일치  
  - `aud` → `EXPECTED_AUDIENCE`와 일치  
  - `exp` / `nbf` → 시간 유효성 검사  
  - `scope` → 필요 시 추가 검증 가능  

### 역방향 프록시
- 토큰이 **유효**하면 `UPSTREAM_URL` 로 요청을 전달하고, 원본 응답을 그대로 반환.  
- 토큰이 **무효**하거나 없으면 `401 Unauthorized`와 JSON 오류 메시지를 반환.  
- 무상태 설계이므로 수평 확장이 용이하고, 쿠버네티스 등 오케스트레이션 환경에서 `ReplicaSet` 으로 배포 가능.

## 7. 설치 및 빠른 시작
### 사전 요구 사항
- Go 1.20 이상 (소스 빌드 시) 또는 Docker 런타임.  
- 인증 서버(OIDC‑준수)와 JWKS 엔드포인트가 준비돼 있어야 함.

### 환경 변수 설정 (예시)
- `LISTEN_ADDR=0.0.0.0:8080` – 프록시가 바인드할 주소·포트  
- `UPSTREAM_URL=http://your-mcp-server:8000` – 실제 MCP 서버 URL  
- `RESOURCE_URI=https://mcp.yourdomain.com` – 메타데이터 베이스 URL  
- `AUTHORIZATION_SERVER=https://auth.yourdomain.com/application/o/mcp/`  
- `JWKS_URI=https://auth.yourdomain.com/application/o/mcp/jwks/`  
- `EXPECTED_ISSUER=https://auth.yourdomain.com/application/o/mcp/`  
- `EXPECTED_AUDIENCE=your-client-id`

### 로컬 실행
```sh
export LISTEN_ADDR=0.0.0.0:8080
export UPSTREAM_URL=http://your-mcp-server:8000
export RESOURCE_URI=https://mcp.yourdomain.com
export AUTHORIZATION_SERVER=https://auth.yourdomain.com/application/o/mcp/
export JWKS_URI=https://auth.yourdomain.com/application/o/mcp/jwks/
export EXPECTED_ISSUER=https://auth.yourdomain.com/application/o/mcp/
export EXPECTED_AUDIENCE=your-client-id
go run ./cmd/mcp-gate
```

### Docker 사용
```sh
docker pull cpremus/mcp-gate:latest
docker run -d \
  -e LISTEN_ADDR=0.0.0.0:8080 \
  -e UPSTREAM_URL=http://your-mcp-server:8000 \
  -e RESOURCE_URI=https://mcp.yourdomain.com \
  -e AUTHORIZATION_SERVER=https://auth.yourdomain.com/application/o/mcp/ \
  -e JWKS_URI=https://auth.yourdomain.com/application/o/mcp/jwks/ \
  -e EXPECTED_ISSUER=https://auth.yourdomain.com/application/o/mcp/ \
  -e EXPECTED_AUDIENCE=your-client-id \
  -p 8080:8080 cpremus/mcp-gate:latest
```

레포지토리에는 `docker-compose.example.yml` 이 포함돼 있어 일반적인 MCP 서버와 함께 연결하는 예시를 제공한다 [[출처](https://euno.news/posts/ko/i-couldnt-find-an-oauth-21-proxy-for-mcp-servers-s-439368)].

## 8. 구성 옵션 상세
| 옵션 | 설명 | 기본값 |
|------|------|--------|
| `LISTEN_ADDR` | 프록시가 수신할 주소·포트 | `0.0.0.0:8080` |
| `UPSTREAM_URL` | 실제 MCP 서버 엔드포인트 | **필수** |
| `RESOURCE_URI` | 메타데이터 베이스 URL (스키마 포함) | **필수** |
| `AUTHORIZATION_SERVER` | 인증 서버 베이스 URL | **필수** |
| `JWKS_URI` | JWKS 공개키 엔드포인트 | **필수** |
| `EXPECTED_ISSUER` | JWT `iss` 검증값 | **필수** |
| `EXPECTED_AUDIENCE` | JWT `aud` 검증값 | **필수** |
| `TLS_CERT_FILE` / `TLS_KEY_FILE` | HTTPS 적용 시 TLS 인증서·키 경로 | 미설정 시 HTTP |

TLS 적용 방법: `TLS_CERT_FILE` 와 `TLS_KEY_FILE` 를 지정하면 프록시가 HTTPS 로 바인드한다.

## 9. 보안 고려사항
- **JWKS 캐시 TTL**: 오래된 키가 남지 않도록 적절히 짧은 TTL(예: 5 분)으로 설정한다.  
- **서명 알고리즘 제한**: 현재 RS256만 허용하도록 구현돼 있다; 필요 시 코드 수정으로 제한을 강화할 수 있다.  
- **오류 응답 최소화**: 인증 실패 시 상세 내부 오류를 노출하지 않으며, 로그에만 기록한다.  
- **민감 정보 로그 방지**: 구조화된 JSON 로그에 토큰 자체를 포함하지 않는다.

## 10. 로깅 및 모니터링
- **로그 포맷**: JSON 형태, 주요 필드 `method`, `path`, `status`, `duration_ms`, `client_ip`, `user_agent`.  
- **로그 레벨**: `INFO`(일반 요청), `WARN`(인증 실패), `ERROR`(시스템 오류) 로 구분 가능; 환경 변수 `LOG_LEVEL` 로 조정.  
- **메트릭**:  
  - 요청 수 (`requests_total`)  
  - 인증 성공 비율 (`auth_success_total`)  
  - 인증 실패 비율 (`auth_failure_total`)  
  - JWKS 조회 횟수 (`jwks_fetch_total`)  

Prometheus exporter가 내장돼 있지는 않지만, `expvar` 혹은 `net/http/pprof` 를 활용해 커스텀 메트릭을 노출할 수 있다.

- **지원 MCP 구현**: Go, Python, TypeScript 등 언어에 관계없이 프록시 앞에 두면 동작.

- **제한 사항**: 현재 RS256 서명만 검증; 다른 알고리즘 필요 시 소스 수정 필요.  
- **알려진 이슈**: JWKS 캐시가 오래된 경우 토큰 검증 실패 – 캐시 TTL 조정으로 해결 가능 [[출처](https://euno.news/posts/ko/i-couldnt-find-an-oauth-21-proxy-for-mcp-servers-s-439368)].

## 13. 확장 및 커스터마이징
- **플러그인·미들웨어**: `mcp-gate` 코드는 Go `http.Handler` 체인 구조이므로, 새로운 미들웨어를 삽입해 로깅, 레이트리밋 등 추가 가능.  
- **커스텀 메타데이터 필드**: `RESOURCE_URI` 기반 JSON에 추가 필드를 삽입하려면 `metadata.go` 를 수정한다.  
- **토큰 검증 로직**: `jwt_validator.go` 에서 클레임 검증 함수를 오버라이드하거나, `EXPECTED_ISSUER`·`EXPECTED_AUDIENCE` 외에 사용자 정의 검증을 구현한다.

## 14. 테스트 및 문제 해결
### 기본 동작 검증 절차
1. **메타데이터 조회**: `curl https://<proxy>/ .well-known/oauth-protected-resource` → RFC 9728 JSON 반환 확인.  
2. **JWT 발급**: 인증 서버에서 클라이언트 ID·시크릿으로 토큰 획득.  
3. **프록시 라우팅**: `curl -H "Authorization: Bearer <jwt>" https://<proxy>/your-mcp-endpoint` → 정상 응답이면 성공.  
4. **무효 토큰 테스트**: 변조된 JWT 로 요청 → 401 반환 확인.

### 일반 오류와 해결 방안
- **404 on metadata endpoint** → `RESOURCE_URI` 혹은 `LISTEN_ADDR` 설정 오류 확인.  
- **401 on valid token** → `EXPECTED_ISSUER`·`EXPECTED_AUDIENCE` 값이 인증 서버와 일치하는지 검증.  
- **JWKS fetch 실패** → `JWKS_URI` 접근 가능 여부와 네트워크 정책 확인.  
- **로그에 토큰이 노출** → 로그 레벨을 `INFO` 이하로 낮추고, `DEBUG` 모드가 활성화돼 있지 않은지 확인.

### 디버깅 팁
- 환경 변수 `LOG_LEVEL=DEBUG` 로 실행하면 JWT 파싱·검증 과정을 상세히 확인할 수 있다.  
- 컨테이너 내부에서 `curl` 로 JWKS 엔드포인트 직접 호출해 응답 형식을 검증한다.

## 15. FAQ
**Q1. 왜 사이드카 프록시가 필요한가?**  
A: MCP 서버는 인증 로직을 제공하지 않으며, Claude.ai 같은 클라이언트는 RFC 9728 메타데이터와 JWT 검증을 전제한다. 사이드카가 이를 대신 구현해 기존 MCP 서버를 그대로 사용할 수 있다.

**Q2. Docker 없이 직접 빌드·실행하려면?**  
A: Go 1.20 이상이 설치된 환경에서 레포지토리 `./cmd/mcp-gate` 디렉터리를 `go run` 혹은 `go build` 후 실행한다. 환경 변수만 올바르게 설정하면 된다.

**Q3. JWKS 캐시가 업데이트되지 않을 때는?**  
A: 캐시 TTL을 짧게(예: 60 초) 설정하거나, `SIGUSR1` 시그널을 보내어 강제 재조회하도록 구현할 수 있다. 현재 구현에서는 TTL 기반 자동 갱신을 제공한다.

## 17. 부록
### 환경 변수 전체 목록
- `LISTEN_ADDR` – 프록시 바인드 주소·포트 (예: `0.0.0.0:8080`)  
- `UPSTREAM_URL` – 실제 MCP 서버 URL (필수)  
- `RESOURCE_URI` – 메타데이터 베이스 URL (필수)  
- `AUTHORIZATION_SERVER` – 인증 서버 베이스 URL (필수)  
- `JWKS_URI` – JWKS 엔드포인트 URL (필수)  
- `EXPECTED_ISSUER` – JWT `iss` 검증값 (필수)  
- `EXPECTED_AUDIENCE` – JWT `aud` 검증값 (필수)  
- `TLS_CERT_FILE` / `TLS_KEY_FILE` – HTTPS 사용 시 인증서·키 파일 경로 (선택)  
- `LOG_LEVEL` – `DEBUG`, `INFO`, `WARN`, `ERROR` 중 선택 (기본 `INFO`)  

### 예시 `docker-compose.yml`
```yaml
version: "3.8"
services:
  mcp-gate:
    image: cpremus/mcp-gate:latest
    environment:
      - LISTEN_ADDR=0.0.0.0:8080
      - UPSTREAM_URL=http://mcp-server:8000
      - RESOURCE_URI=https://mcp.example.com
      - AUTHORIZATION_SERVER=https://auth.example.com/application/o/mcp/
      - JWKS_URI=https://auth.example.com/application/o/mcp/jwks/
      - EXPECTED_ISSUER=https://auth.example.com/application/o/mcp/
      - EXPECTED_AUDIENCE=client-id
    ports:
      - "8080:8080"
  mcp-server:
    image: your-mcp-image:latest
    ports:
      - "8000:8000"
```

### 인증 서버 설정 예시 (Keycloak)
1. 클라이언트 생성 → `client-id` 와 `client-secret` 확보.  
2. 클라이언트에 `Bearer Only` 옵션 활성화.  
3. `Authorization Endpoint` 와 `JWKS URL`을 `mcp-gate` 환경 변수에 지정.  

> 위 Keycloak 설정 예시는 일반적인 절차이며, 실제 값은 인증 서버에 따라 다릅니다. 추가 조사가 필요합니다.
