## 개요
- **취약점 ID**: CVE‑2026‑27574 (GHSA‑H343‑GG57‑2Q67)  
- **CVSS 점수**: 10.0 (Critical) – 네트워크 기반 원격 코드 실행 가능 [출처: euno.news]  
- **공개 일자**: 2026‑03‑07 [출처: euno.news]  
- **현재 상태**: PoC(Proof‑of‑Concept) 존재, 공격 코드 공개 [출처: euno.news]  

## OneUptime Probe 소개
OneUptime은 클라우드 기반 모니터링 플랫폼으로, **Synthetic Monitor / Probe** 컴포넌트를 통해 사용자가 정의한 스크립트를 실행해 서비스 가용성을 검사한다.  
- **구성 요소**: Probe, API 서버, 데이터베이스, Redis, ClickHouse 등  
- **보안 설계**: 기존에는 Node.js 표준 `node:vm` 모듈을 이용해 사용자 스크립트를 격리된 VM(가상 머신) 환경에서 실행하도록 구현했다.  

## 취약점 상세 분석
- **발생 원리**: `node:vm` 모듈은 완전한 격리를 제공하지 못한다. 공격자는 스크립트 내부에서 `this.constructor.constructor` 를 호출해 `Function` 객체에 접근하고, 이를 이용해 호스트 프로세스의 컨텍스트로 탈출한다.  
- **관련 CWE**: CWE‑94 (코드 인젝션) [출처: euno.news]  
- **영향받는 버전**: OneUptime **≤ 9.5.13**  
- **수정된 버전**: **10.0.5** – `node:vm` 사용을 중단하고, 보다 안전한 격리 라이브러리인 **isolated‑vm** 로 마이그레이션 [출처: euno.news]  
- **코드 변경**: 커밋 `7f9ed4d`에서 `SyntheticMonitor/Worker.ts` 파일의 `node:vm` import와 사용을 제거하고, `isolated‑vm` 로 교체함 [출처: euno.news]  

## 공격 시나리오
1. **전제 조건**: 공격자는 프로젝트 내 **Project Member** 권한을 보유하고 있어 Probe에 사용자 정의 스크립트를 업로드할 수 있다 [출처: euno.news].  
2. **공격 벡터**: 네트워크를 통해 악성 스크립트를 Probe에 전달한다 (Attack Vector: Network) [출처: euno.news].  
3. **샌드박스 탈출**: 스크립트 내부에서 `this.constructor.constructor` 를 이용해 `Function` 객체를 획득, 호스트 Node.js 프로세스에서 임의 코드를 실행한다.  
4. **탈취 가능한 자산**: 전체 클러스터 비밀(데이터베이스 비밀번호, Redis 토큰, `ONEUPTIME_SECRET` 등) 및 시스템 명령 실행 권한을 획득한다 [출처: euno.news].  

## 영향 평가
- **시스템 전반 위험**: 클러스터 전체 비밀이 노출되어 데이터베이스, 캐시, 로그 등 모든 서비스가 침해될 위험이 있다.  
- **가용성**: 공격자는 서비스 중단(RCE를 통한 프로세스 종료)도 가능하므로 가용성에 심각한 위협을 가한다.  
- **데이터 무결성**: 비밀이 탈취되면 데이터베이스 조작·삭제·변조가 가능하다.  
- **EPSS 점수**: 0.00055 (현재 활용 가능성은 낮지만, PoC가 존재함) [출처: euno.news]  

## 탐지 및 대응 방안
### 로그·모니터링 포인트
- Probe 컨테이너의 **stderr/stdout**에 비정상적인 `Function` 객체 호출 로그  
- `node:vm` 모듈 사용 흔적이 남아 있는 오래된 이미지 실행 시도  
- Synthetic Monitor 스크립트 업로드 시 **User‑Agent** 혹은 **IP**가 평소와 다른 경우  

### 이상 행동 탐지 규칙 예시
- “Synthetic Monitor 스크립트 실행 중 `this.constructor.constructor` 문자열이 포함된 경우” 알림  
- “Probe 컨테이너가 5분 이내에 10회 이상 재시작” 경보  

### 임시 방어 조치
- **Synthetic Monitor 기능 비활성화**(관리 콘솔 → Settings → Synthetic Monitor → Disable) [출처: euno.news]  
- **Project Member** 권한을 최소화하고, 신뢰할 수 있는 사용자만 허용  

## 완화 및 패치 적용 방법
1. **Docker 이미지 업데이트**  
   - 최신 OneUptime 이미지(`10.0.5` 이상) Pull: `docker pull oneuptime/oneuptime:10.0.5`  
2. **서비스 재배포**  
   - Probe와 API 서버를 새로운 이미지로 재시작하고, 기존 컨테이너를 교체한다.  
3. **클러스터 비밀 교체**  
   - PostgreSQL, Redis, ClickHouse 비밀번호를 새로 생성하고, 환경 변수 `ONEUPTIME_SECRET` 를 업데이트한다.  
4. **전체 클러스터 재시작**  
   - 모든 노드에서 서비스 재시작 후 정상 동작 여부 확인.  

## 사후 복구 절차
- **시크릿 재생성**: 데이터베이스·Redis·ClickHouse 비밀번호와 `ONEUPTIME_SECRET` 를 모두 새로 발급한다.  
- **클러스터 전체 재시작**: 모든 서비스가 새로운 시크릿을 사용하도록 강제 재시작한다.  
- **모니터링 스크립트 검증**: 기존 사용자 정의 스크립트를 하나씩 재배포해 정상 동작을 확인한다.  
- **감사 로그 검토**: 공격 기간 동안의 Probe 로그를 분석해 추가 침해 흔적이 없는지 확인한다.
