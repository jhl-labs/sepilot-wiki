## 개요  
- **Joy 프로젝트**는 AI 에이전트 간에 **AI‑to‑AI 인증·보증(vouching)** 을 가능하게 하는 오픈 트러스트 네트워크를 제공한다. 에이전트가 다른 에이전트를 보증함으로써 인간 개입 없이도 행동을 위임할 수 있다【euno.news】.  
- **AI‑to‑AI 인증·보증**은 복잡한 자동화 파이프라인에서 인간 승인 단계가 병목이 되는 문제를 해결하고, 빠른 의사결정을 지원한다.  
- **대상 독자**: AI 플랫폼 엔지니어, DevOps 팀, 보안 담당자, 그리고 AI‑driven 워크플로우를 설계·운영하는 개발자.  

## 배경 및 동기  
- 기존 AI 파이프라인은 **인간 승인**(예: PR 리뷰, 결제 승인) 단계가 필수이지만, 이는 지연과 운영 비용을 초래한다.  
- **신뢰 체인**과 **머신‑서명 클레임** 개념을 도입하면, 에이전트가 자체적으로 검증 가능한 증명을 전달해 downstream 에이전트가 자동으로 신뢰할 수 있다.  
- 유사한 목적을 가진 기존 표준·프로토콜으로는 **OAuth 2.0**, **JWT**, **SPIFFE**(Secure Production Identity Framework For Everyone) 등이 있다. 각각의 공식 문서는 아래와 같다.  
  - OAuth 2.0: https://datatracker.ietf.org/doc/html/rfc6749  
  - JWT: https://datatracker.ietf.org/doc/html/rfc7519  
  - SPIFFE: https://spiffe.io  

## 핵심 개념 정의  
- **에이전트**: 자동화된 작업을 수행하는 AI 프로그램(예: 코드 생성, 데이터 쿼리, 결제 트리거).  
- **보증(vouching) 에이전트**: 다른 에이전트를 **머신‑서명된 클레임**으로 보증하여, 해당 에이전트가 수행하는 행동을 downstream 에이전트가 신뢰하도록 하는 주체.  
- **머신‑서명 클레임**: 에이전트가 자신의 비밀키로 서명한 JSON‑형식의 증명. 클레임에는 **발행자**, **대상**, **스코프**, **TTL(Time‑to‑Live)** 등이 포함된다.  
- **단기/취소 가능한 증명**: TTL이 짧고, 발행자가 언제든지 **폐기(revoke)** 할 수 있는 형태의 클레임.  
- **신뢰 네트워크 토폴로지**  
  - **다이렉트**: 보증 에이전트가 직접 downstream 에이전트에 클레임을 전달.  
  - **트랜시티브**: 보증 체인이 여러 단계로 이어져, 최종 에이전트는 전체 체인을 검증한다.  

## 시스템 아키텍처  
### 구성 요소  
1. **에이전트** – 작업을 수행하고 클레임을 생성·전파.  
2. **인증 서버** – 에이전트 공개키를 저장하고, 클레임 검증 API를 제공.  
3. **검증 모듈** – downstream 에이전트가 수신한 클레임을 서명·스코프·TTL을 검사.  
4. **감사 로그** – **append‑only** 형태로 모든 vouching·폐기·승인 이벤트를 기록.  

### 데이터 흐름 및 메시지 시퀀스  
1. 보증 에이전트가 **키 쌍**을 생성하고 인증 서버에 **공개키**를 등록.  
2. 작업 요청 시 보증 에이전트는 **클레임**을 서명하고 downstream 에이전트에 전송.  
3. downstream 에이전트는 인증 서버에 서명 검증을 요청하고, 스코프·TTL을 확인 후 작업을 실행.  
4. 키가 폐기되면 인증 서버는 해당 키를 **revoked list**에 추가하고, 이후 모든 클레임을 거부한다.  

### API 인터페이스 (예시)  
- `POST /v1/claims` – 클레임 생성 (보증 에이전트)  
- `GET /v1/claims/{id}` – 클레임 조회·검증 (downstream)  
- `POST /v1/keys/revoke` – 키 폐기 요청 (관리자)  
- `GET /v1/audit` – 감사 로그 스트리밍 (읽기 전용)  

> **주의**: 실제 API 스펙은 프로젝트 레포지토리와 인증 서버 구현에 따라 달라질 수 있다. 추가 조사가 필요합니다.  

## 신뢰 모델 및 인증 흐름  
1. **클레임 생성** – 보증 에이전트는 자신의 비밀키로 JSON 클레임에 서명하고, `iss`(issuer), `sub`(subject), `scope`, `exp`(expiration) 등을 포함한다.  
2. **전파** – 서명된 클레임은 메시지 큐, HTTP 헤더, 혹은 파일 메타데이터 등으로 downstream 에이전트에 전달된다.  
3. **검증** – downstream 에이전트는 인증 서버에 `GET /v1/claims/{id}` 요청을 보내 서명을 검증하고, 스코프가 현재 액션에 허용되는지 확인한다.  
4. **권한 위임** – 검증이 성공하면, downstream 에이전트는 **위임된 권한**(예: PR 생성, SQL 실행)으로 작업을 수행한다.  
5. **폐기·갱신** – 키가 폐기되면 인증 서버는 즉시 해당 키를 거부하고, 클레임 TTL이 만료되면 자동으로 무효화한다.  

## 보안 및 위험 관리  
### 주요 위협 시나리오  
- **키 유출**: 공격자가 보증 에이전트의 비밀키를 탈취하면, 악의적인 클레임을 생성해 대규모 자동화 공격을 수행할 수 있다.  
- **권한 상승**: 과도한 스코프가 부여된 클레임은 downstream 에이전트가 의도치 않은 작업을 수행하게 만든다.  
- **블라스트 반경 확대**: 하나의 보증 에이전트가 여러 downstream 에이전트를 제어하면, 해당 에이전트가 손상될 경우 영향 범위가 급격히 확대된다.  

### Practical Controls (Joy 프로젝트가 제안)  
- **BYOK (Bring‑Your‑Own‑Key)** – 각 조직이 자체 키를 관리하고, 인증 서버에 공개키만 등록한다.  
- **Mandatory Human Approval** – 중요한 publish/transaction 단계에서 인간 승인을 강제한다.  
- **Per‑Action Scopes** – 클레임마다 세밀한 스코프를 정의해 최소 권한 원칙을 적용한다.  
- **Short‑Lived Signed Claims** – TTL을 짧게 설정해 클레임 유효 기간을 제한한다.  
- **Revocable Attestations** – 키 폐기 시 즉시 모든 기존 클레임을 무효화한다.  
- **Append‑Only Audit Logs** – 모든 vouching·폐기·승인 이벤트를 변경 불가능한 로그에 기록한다.  

## 구현 가이드  
### 개발 환경 설정  
1. **언어/런타임**: Node.js, Python, Go 등 원하는 언어에서 JWT/JOSE 라이브러리를 사용한다.  
2. **의존성** (예시)  
   - Node.js: `npm install jsonwebtoken jose`  
   - Python: `pip install pyjwt cryptography`  
   - Go: `go get github.com/dgrijalva/jwt-go`  

### 에이전트 등록·키 관리 절차  
1. 에이전트는 **RSA/ECDSA** 키 쌍을 생성한다 (`openssl genpkey -algorithm RSA -out private.pem`).  
2. 공개키(`public.pem`)를 인증 서버에 `POST /v1/keys` 로 등록한다.  
3. 키 회전 정책에 따라 **주기적 키 교체**와 **폐기** 절차를 문서화한다.  

### Vouching 에이전트 구현 예시 (인라인 코드)  
- **클레임 서명** (Python)  
  ```python
  import jwt, datetime
  private_key = open('private.pem').read()
  claim = {
      "iss": "agent-A",
      "sub": "agent-B",
      "scope": ["github:pull_request:create"],
      "exp": datetime.datetime.utcnow() + datetime.timedelta(minutes=5)
  }
  signed = jwt.encode(claim, private_key, algorithm="RS256")
  ```  
- **클레임 검증** (Node.js)  
  ```javascript
  const jwt = require('jsonwebtoken');
  const publicKey = fs.readFileSync('public.pem');
  const decoded = jwt.verify(signedToken, publicKey, { algorithms: ['RS256'] });
  // scope 검사 후 작업 수행
  ```  

> **주의**: 위 코드는 개념 증명용이며, 실제 서비스에서는 **키 보호**, **오디트 로깅**, **에러 처리** 등을 추가해야 한다.  

### CI/CD 파이프라인 통합  
1. **빌드 단계**에서 보증 에이전트가 자동으로 클레임을 생성하고, GitHub Actions 등 워크플로에 전달한다.  
2. **배포 단계**에서는 downstream 에이전트가 클레임을 검증하고, 승인된 경우에만 배포 스크립트를 실행한다.  
3. **Human‑in‑the‑Loop** 단계는 `approval` 워크플로(예: GitHub Environments)와 연동해 필수 승인을 강제한다.  

## 테스트 및 검증 방법  
### 10분 내에 수행 가능한 **Test to Run**  
1. **Vouching 에이전트 생성** – 위 “키 관리 절차”에 따라 에이전트를 만들고 클레임을 발행한다.  
2. **키 폐기** – 인증 서버에 `POST /v1/keys/revoke` 로 해당 에이전트의 비밀키를 폐기한다.  
3. **Downstream 동작 확인** – downstream 에이전트가 여전히 보증된 작업을 수행하는지 관찰한다. 정상적인 경우 폐기된 키의 클레임은 즉시 거부되어 작업이 중단돼야 한다.  
4. **감지 시간 측정** – 폐기 이벤트가 로그에 기록되고, downstream이 이를 인식하는 데 걸린 시간을 측정한다.  

### 자동화 테스트 스크립트 예시 (Bash)  
```bash
# 1. 발행
TOKEN=$(python gen_claim.py)   # 클레임 생성 스크립트
# 2. 폐기
curl -X POST https://auth.example.com/v1/keys/revoke -d '{"kid":"agent-A-key"}'
# 3. 검증
curl -H "Authorization: Bearer $TOKEN" https://downstream.example.com/execute
# 기대 결과: 403 Forbidden
```  

> **주의**: 실제 환경에서는 위 스크립트를 CI 파이프라인에 통합하고, 결과를 **audit log**와 비교해 자동화된 검증을 수행한다.  

## 활용 사례  
| 분야 | 시나리오 | 기대 효과 |
|------|----------|-----------|
| **코드 리뷰 자동화** | Claude → GitHub PR 생성 파이프라인에서 Joy를 사용해 AI가 PR을 자동 생성하고, 보증 에이전트가 인간 승인 없이도 PR을 머지 | 리뷰 속도 30%‑50% 향상 |
| **데이터베이스 자동 쿼리** | AI가 분석 결과에 따라 자동으로 SQL 쿼리를 실행, 보증 에이전트가 `db:write` 스코프를 부여 | 데이터 파이프라인 지연 감소 |
| **결제·재무 트랜잭션** | AI가 비용 청구를 자동 생성, 보증 에이전트가 `payments:execute` 스코프를 부여 | 인보이스 처리 시간 단축 |
| **법률·규제 환경** | 금융 서비스에서 AI가 계약 초안을 자동 생성, 인간 검토를 UI 레이어에서 강제(제품 기능) | 규제 준수와 자동화 균형 |

## 베스트 프랙티스  
- **최소 권한 원칙**: 각 클레임에 필요한 최소 스코프만 부여하고, 불필요한 권한은 제거한다.  
- **키 회전 정책**: 30‑90일 주기로 키를 교체하고, 폐기된 키는 즉시 인증 서버에 반영한다.  
- **감사 로그 보관**: 로그는 **append‑only** 스토리지(예: CloudWatch Logs, Elasticsearch) 에 최소 1년간 보관한다.  
- **Human Approval UI/UX**: 승인 화면에 클레임 상세(issuer, scope, TTL)를 명시하고, “승인” 버튼을 클릭하기 전 반드시 검토하도록 설계한다.  

## 향후 로드맵 및 확장성  
- **멀티‑클라우드·멀티‑테넌시** 지원을 위해 인증 서버를 **Kubernetes** 기반으로 배포하고, 네임스페이스 별 키 관리 정책을 적용할 계획이다.  
- **표준화 작업**: SPIFFE와 연계해 **SPIFFE ID** 기반 에이전트 식별자를 도입하고, 커뮤니티와 협업해 OIDC‑compatible 토큰 포맷을 제안한다.  
- **규제 대응**: 금융·헬스케어 분야에서 요구되는 **감사 추적**과 **인증서 기반 서명** 요구사항을 충족하도록 정책 엔진을 강화한다.  

## 결론  
Joy는 **AI‑to‑AI 인증·보증**을 통해 인간 승인 없이도 안전하게 자동화 파이프라인을 운영할 수 있게 한다. 핵심 가치는 **빠른 파이프라인**과 **위험 최소화**이며, **BYOK**, **단기 클레임**, **취소 가능한 증명**, **감사 로그** 등 실용적인 보안 제어를 제공한다. 도입 시에는 **키 관리**, **스코프 설계**, **인간 승인 UI** 등을 신중히 설계해야 하며, 실제 운영 환경에서는 추가적인 검증과 모니터링이 필요하다.
