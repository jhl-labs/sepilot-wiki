## 소개
AI 에이전트는 이제 연구 실험을 넘어 실제 프로덕션 서비스와 DevOps 파이프라인을 자동화하는 핵심 컴포넌트로 자리 잡았습니다. 본 가이드는 **“LLM을 호출하고 응답을 출력한다”** 수준에 머무는 기존 튜토리얼을 넘어, **추론·툴 사용·세션 컨텍스트 기억·다중 에이전트 확장**까지 지원하는 완전한 파이썬 에이전트를 처음부터 끝까지 구현하고 배포하는 과정을 단계별로 제시합니다.  

- **목표**: 로컬에서 에이전트를 개발·디버깅하고, GitHub Actions 기반 CI/CD 파이프라인을 구축해 스테이징·프로덕션에 안전하게 배포한다.  
- **대상 독자**: Python 기본 문법, Git, Docker 정도의 사전 지식이 있는 개발자. LLM API 사용 경험이 있으면 더욱 원활합니다.  
- **출처**: 원문 기사 “첫 AI 에이전트를 만드는 방법: 단계별 튜토리얼”([euno.news](https://euno.news/posts/ko/how-to-build-your-first-ai-agent-a-step-by-step-tu-d97ba7)) 에서 핵심 내용 발췌.

## Agent vs. Chatbot
| 구분 | Chatbot | AI Agent |
|------|---------|----------|
| **대화 상태** | 무상태 혹은 약간의 상태 유지 | 지속적인 상태와 메모리 관리 |
| **행동 범위** | 텍스트 응답만 | 파일·API·코드 실행 등 실제 행동 |
| **핵심 기능** | `Perceive`·`Reason`·`Act`·`Remember` 중 **Act** 미구현 | 네 가지 모두 구현 (Perceive, Reason, Act, Remember) |
| **예시** | “날씨 알려줘” → 텍스트 반환 | “날씨 API 호출 → 결과 저장 → 다음 작업 계획” |

### Perceive, Reason, Act, Remember
- **Perceive**: 파일 읽기, API 조회, 로그 관찰 등 환경 인식.  
- **Reason**: 목표 달성을 위한 계획·자기 성찰.  
- **Act**: 코드 실행, API 호출, 파일 쓰기 등 실제 행동.  
- **Remember**: 단기 스크래치패드와 장기 저장소에 상태 저장.  

가장 단순한 사고 루프는 아래와 같은 **while‑loop** 형태입니다.

```
while not goal_achieved:
    observe_environment()
    think_next_step()
    take_action()
    record_result()
```

## 핵심 아키텍처
```
+------------------------------------------------------+
|                    AGENT CORE                         |
|   +------------+    +------------+    +------------+  |
|   |    LLM     |    |   TOOLS    |    |   MEMORY   |  |
|   +------------+    +------------+    +------------+  |
|          |                |                |          |
|          v                v                v          |
|   +----------------------------------------------+   |
|   |          PLANNING / ORCHESTRATION LAYER       |   |
|   +----------------------------------------------+   |
+------------------------------------------------------+
```
- **LLM**: Claude, GPT‑4, Llama 3 등 다단계 지시를 신뢰성 있게 수행하는 모델.  
- **Tools**: 모델이 호출할 수 있는 함수·API (예: 파일 I/O, 외부 서비스). MCP·OpenAI 함수 호출 사양을 활용합니다.  
- **Memory**: 단기 스크래치패드와 장기 DB(예: SQLite, Redis)로 구성.  
- **Planning / Orchestration**: LLM 출력 → Tool 호출 → Memory 업데이트 순서 제어.

## 프로젝트 구조와 기본 에이전트 설계
```
my_agent/
├─ agent_core.py          # 메인 루프와 Orchestration
├─ config/
│   └─ settings.yaml      # 모델·툴·시크릿 설정
├─ toolkit/
│   ├─ __init__.py
│   ├─ file_io.py
│   └─ api_client.py
├─ memory/
│   ├─ __init__.py
│   ├─ short_term.py
│   └─ long_term.py
├─ tests/
│   ├─ test_core.py
│   └─ test_toolkit.py
└─ pyproject.toml        # Poetry 의존성 정의
```

1. **설정**: `config/settings.yaml` 에 LLM 엔드포인트, API 키, 툴 메타데이터를 선언합니다.  
2. **핵심 루프** (`agent_core.py`):
   - 설정 로드 → 목표 정의 → `while not goal` 루프 실행  
   - `perceive()` → `reason()` → `act()` → `remember()` 순서 호출  
3. **툴 구현**: `toolkit/file_io.py` 에 `read_file`, `write_file` 함수, `api_client.py` 에 REST 호출 래퍼를 작성합니다.  
4. **메모리**: `memory/short_term.py` 에 파이썬 딕셔너리 기반 스크래치패드, `memory/long_term.py` 에 SQLite 혹은 Redis 연동 로직을 구현합니다.

## CI/CD 파이프라인 설정 (GitHub Actions)
`.github/workflows/ci.yml`

```yaml
name: CI
on:
  push:
    branches: [ main ]
jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: "3.11"
      - name: Install Poetry
        run: curl -sSL https://install.python-poetry.org | python3 -
      - name: Install dependencies
        run: poetry install --no-interaction --no-root
      - name: Lint
        run: poetry run flake8 .
      - name: Test
        run: poetry run pytest
      - name: Build Docker image
        run: |
          docker build -t my-agent:${{ github.sha }} .
      - name: Push to registry
        if: github.ref == 'refs/heads/main'
        run: |
          echo "${{ secrets.REGISTRY_PASSWORD }}" | docker login -u ${{ secrets.REGISTRY_USER }} --password-stdin
          docker push my-agent:${{ github.sha }}
```

- **캐시·배치 전략**: `actions/cache` 로 Poetry 패키지와 Docker 레이어 캐시를 저장해 추론 비용을 최소화합니다.  
- **시크릿 관리**: GitHub Secrets (`OPENAI_API_KEY`, `REGISTRY_PASSWORD` 등) 에 저장하고 `env` 로 전달합니다.

## 다중 에이전트 시스템으로 확장
1. **협업 패턴**  
   - **마스터‑워커**: 중앙 마스터가 목표를 분할하고 워커에게 작업 할당.  
   - **브로드캐스트**: 모든 에이전트가 동일 목표를 공유하고 독립적으로 진행 후 결과 합산.  

2. **공유 메모리·메시징**  
   - **Redis Pub/Sub** 혹은 **RabbitMQ** 로 작업 큐와 상태 전파 구현.  
   - 각 에이전트는 `memory/long_term.py` 를 통해 공통 DB에 결과 저장.  

3. **성능·보안 고려사항**  
   - **성능**: 툴 호출 시 비동기 I/O (`asyncio`) 적용, 배치 API 사용.  
   - **보안**: 시크릿은 환경 변수 혹은 Vault에 저장, 툴 호출 권한 최소화.  

## 베스트 프랙티스 & 흔히 발생하는 문제
| 주제 | 팁 | 흔한 오류 & 해결 |
|------|-----|-------------------|
| **프롬프트 엔지니어링** | 시스템 프롬프트에 `Perceive → Reason → Act → Remember` 흐름을 명시 | LLM이 툴 호출을 건너뛰면 `function_call` 파라미터를 강제 지정 |
| **비용 최적화** | 배치 추론, 결과 캐시, 토큰 제한 관리 | 과도한 토큰 사용 시 `RateLimitError` → 토큰 수 제한 로직 추가 |
| **디버깅** | 각 단계별 로그 레벨 구분 (`PERCEIVE`, `REASON`, `ACT`, `REMEMBER`) | 무한 루프 발생 → 목표 달성 조건(`goal_achieved`) 검증 로직 추가 |
| **테스트** | Mock LLM 응답 (`unittest.mock`) 으로 툴 호출 검증 | 외부 API 호출 실패 → `responses` 라이브러리로 모킹 |
