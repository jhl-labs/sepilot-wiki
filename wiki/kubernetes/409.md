## 1. 서론
마이크로서비스 아키텍처가 확장됨에 따라 각 서비스의 엔드포인트, 포트, 네트워킹 요구 사항을 일일이 관리하기가 어려워집니다. 서비스 간 의존성, 동적 스케일링, 이기종(온프레미스·클라우드·하이브리드) 환경, 다중 버전 운영 등은 **서비스 디스커버리**가 핵심 인프라가 되어야 함을 의미합니다. 본 문서는 쿠버네티스가 제공하는 기본 메커니즘과 서비스 메시(Istio 등)를 활용한 고급 라우팅을 정리하여, 마이크로서비스 운영자가 안정적인 디스커버리를 설계·운영할 수 있도록 돕는 것을 목표로 합니다.

## 2. 마이크로서비스 디스커버리 핵심 과제
| 과제 | 설명 |
|------|------|
| **서비스 상호 의존성** | 서비스가 서로를 찾지 못하면 연쇄 장애가 발생합니다. 신뢰할 수 있는 위치 확인과 라우팅 보장이 필요합니다. |
| **동적 스케일링** | 새로운 파드가 추가·제거될 때 엔드포인트 정보를 실시간으로 업데이트해야 합니다. |
| **하이브리드·멀티클라우드** | 온프레미스와 퍼블릭 클라우드에 걸친 서비스가 일관된 방식으로 발견되어야 합니다. |
| **다중 버전 운영** | Canary, Blue/Green 등 여러 버전을 동시에 운영하면서 트래픽을 원하는 버전으로 라우팅해야 합니다. |

## 3. 쿠버네티스 기본 개념
- **클러스터**: 마스터 노드와 워커 노드가 결합된 논리적 집합.  
- **파드(Pod)**: 하나 이상의 컨테이너를 포함하는 최소 배포 단위.  
- **레이블 & 셀렉터**: 파드에 메타데이터를 붙여 논리적 그룹을 정의하고, Service 등에서 해당 그룹을 선택합니다.  
- **네트워킹 모델**  
  - **ClusterIP**: 클러스터 내부 전용 가상 IP. 기본 Service 타입.  
  - **NodePort**: 각 노드의 고정 포트를 통해 외부 접근 가능.  
  - **LoadBalancer**: 클라우드 제공자의 로드밸런서를 자동 프로비저닝.  

## 4. 쿠버네티스 Service 리소스
Service는 동일한 기능을 수행하는 파드들의 논리적 집합을 나타내며, 클라이언트에게 안정적인 네트워크 엔드포인트를 제공합니다. 기본 구조는 다음과 같습니다.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  selector:
    app: my-app
    tier: frontend
  ports:
    - port: 80
      targetPort: 8080
```

*위 예시에서 `my-service`는 `app=my-app` 및 `tier=frontend` 라벨을 가진 파드에 트래픽을 로드밸런싱합니다*【euno.news】.

### Service 타입
- **ClusterIP** (기본) – 클러스터 내부에서만 접근.  
- **Headless** (`clusterIP: None`) – DNS 레코드가 파드 IP 리스트를 직접 반환, StatefulSet 등에 활용.  
- **ExternalName** – 외부 DNS 이름을 내부 Service 이름으로 매핑.  

## 5. 쿠버네티스 내 서비스 디스커버리 메커니즘
| 메커니즘 | 동작 방식 | 장점 | 단점 |
|----------|-----------|------|------|
| **DNS 기반** | `kube-dns`/`CoreDNS`가 Service 이름을 DNS 레코드로 제공. 파드에서 `my-service.namespace.svc.cluster.local` 형태로 조회. | 표준화, 자동 업데이트, 대부분의 애플리케이션에서 기본 지원. | DNS 캐시가 오래 남을 경우 최신 엔드포인트 반영 지연 가능. |
| **환경 변수** | 파드 생성 시 Service IP와 포트를 환경 변수(`MY_SERVICE_SERVICE_HOST`, `MY_SERVICE_SERVICE_PORT`)로 주입. | 간단한 스크립트·배시 기반 서비스에 유용. | 서비스 수가 많을 경우 환경 변수 폭증, 동적 업데이트 불가. |
| **Kubernetes API** | 파드가 `Endpoints` 혹은 `EndpointSlice` 객체를 직접 조회. | 상세 메타데이터(라벨, 토폴로지) 활용 가능, 고급 로직 구현. | API 호출 오버헤드, 인증·RBAC 설정 필요. |

## 6. 동적 스케일링과 자동 업데이트
- **파드 추가·제거** 시 kube‑proxy가 Service의 `Endpoints` 객체를 자동 갱신하고, DNS 레코드도 즉시 반영됩니다.  
- **Horizontal Pod Autoscaler (HPA)** 와 연계하면 CPU·메모리 등 메트릭 기반으로 파드 수가 변동될 때 Service 엔드포인트가 실시간으로 업데이트됩니다.  
- **StatefulSet**·**DaemonSet** 등 특수 워크로드는 Headless Service와 `EndpointSlice`를 활용해 파드별 고정 DNS 이름을 제공할 수 있습니다【euno.news】.

## 7. 서비스 버전 관리와 트래픽 라우팅 전략
1. **라벨·셀렉터 기반 버전 구분**  
   - `app=my-app, version=v1` 과 `app=my-app, version=v2` 라벨을 각각의 파드에 부여하고, 별도 Service(`my-app-v1`, `my-app-v2`)를 정의합니다.  
2. **Canary / Blue‑Green 배포**  
   - 새 버전 파드를 기존 Service에 추가하고, `kubectl rollout pause`/`resume` 혹은 `kubectl set image` 로 트래픽 비율을 조정합니다.  
3. **Ingress·Service Mesh 활용**  
   - Ingress 컨트롤러(NGINX, Traefik 등)에서 `path`·`host` 기반 라우팅 규칙을 정의하거나, Istio VirtualService에서 `weight` 로 트래픽 셰이딩을 구현합니다【euno.news】.

## 8. 서비스 메시(Istio 등)와 고급 라우팅
- **ServiceEntry**: 클러스터 외부 서비스를 Service 객체처럼 선언.  
- **VirtualService**: 요청 매칭 규칙(헤더, URI, 쿠키 등)과 라우팅 대상(DestinationRule) 지정.  
- **DestinationRule**: 서킷 브레이커, 연결 풀, mTLS 설정 등 트래픽 정책 정의.  

예시 (Istio YAML, 코드 블록 없이):
```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: my-app
spec:
  hosts:
    - my-app.namespace.svc.cluster.local
  http:
    - route:
        - destination:
            host: my-app
            subset: v1
          weight: 80
        - destination:
            host: my-app
            subset: v2
          weight: 20
```
*위와 같이 weight 를 조정해 Canary 배포가 가능*【euno.news】.

**베스트 프랙티스**
- Service와 ServiceEntry을 명확히 구분하고, 라벨 네이밍을 일관되게 유지.  
- DNS 기반 디스커버리를 기본으로 하고, 필요 시 EnvVar·API 보조 메커니즘을 활용.  
- CI/CD 파이프라인에 `kubectl get svc, endpoints` 검증 단계와 Istio `istioctl analyze` 를 포함.

## 9. 보안 및 인증
- **mTLS**: Istio가 제공하는 자동 인증으로 서비스 간 트래픽을 TLS로 암호화하고, 양쪽 인증을 수행합니다.  
- **NetworkPolicy**: 네임스페이스·라벨 기반으로 인바운드·아웃바운드 트래픽을 제한해 서비스 디스커버리 단계에서도 최소 권한 원칙을 적용합니다.  
- **RBAC**: 파드가 Kubernetes API를 조회할 경우, `get`, `list` 권한만 부여하도록 Role/ClusterRole을 설계합니다.

## 10. 모니터링·로깅·트러블슈팅
- **Service/Endpoint 상태 확인**  
  - `kubectl get svc my-service -o wide`  
  - `kubectl get endpoints my-service -o yaml`  
- **DNS 조회 문제**  
  - 파드 내부에서 `nslookup my-service.namespace.svc.cluster.local` 실행 후 응답 IP와 TTL 확인.  
- **Prometheus·Grafana**  
  - `kube_service_info`, `kube_endpoint_address` 메트릭을 활용해 서비스 가용성 및 엔드포인트 변동을 시각화.  
- **Istio**: `istioctl proxy-status` 로 Envoy 프록시 상태 확인, `istioctl pc endpoints` 로 DestinationRule 적용 엔드포인트 확인.

## 11. 베스트 프랙티스 요약
1. **라벨 설계** – `app`, `tier`, `version` 등 핵심 라벨을 사전에 정의하고, 네이밍 컨벤션을 문서화합니다.  
2. **DNS 우선** – 기본 디스커버리는 DNS(`ClusterIP`)를 사용하고, 환경 변수는 보조 용도로만 활용합니다.  
3. **자동화** – CI/CD 파이프라인에 `kubectl apply --dry-run=client -f` 검증, `helm test` 혹은 `kustomize` 로 Service 정의 테스트를 포함합니다.  
4. **서비스 메시** – 복잡한 트래픽 제어가 필요할 경우 Istio와 같은 Service Mesh를 도입하고, `VirtualService`·`DestinationRule` 로 선언적 라우팅을 구현합니다.  

## 12. 결론
쿠버네티스는 **Service**, **DNS**, **EndpointSlice** 등 내장 메커니즘을 통해 마이크로서비스 디스커버리의 핵심 과제를 자동화합니다. 이를 기반으로 서비스 메시(Istio)와 Ingress를 결합하면 동적 라우팅, 트래픽 셰이딩, A/B 테스트 등 고급 시나리오도 선언적으로 구현할 수 있습니다. 앞으로는 **Service Mesh 표준화**와 **서버리스·Event‑Driven** 환경과의 통합이 진행되면서, 디스커버리 모델이 더욱 유연해질 전망입니다【euno.news】.
